# AGENTS.md 最佳实践（28条）

> 本文档汇总了编写高质量 AGENTS.md 的 28 条最佳实践，分为**结构**、**内容**、**格式**三个维度。

---

## 一、结构维度最佳实践（10条）

### SP-1: 必须有 YAML 前言区

每个 AGENTS.md 都应以 YAML frontmatter 开头，包含元数据。

```yaml
---
name: project-name-agents
version: v1.0.0
author: author-name
description: 一句话描述这是什么
tags: [tag1, tag2, tag3]
---
```

**为什么重要**:
- Agent 启动时首先解析前言区，快速建立上下文
- 支持自动化工具的元数据提取
- 便于版本管理和索引

**反面案例**:
❌ 直接以 `# 项目说明` 开头，无任何元数据

---

### SP-2: 明确触发条件

清晰定义"何时应该使用这个 AGENTS.md"，而非模糊表述。

✅ **正确**:
```markdown
## 激活条件
当用户提及以下关键词时激活：
- "MoonBit"、"mbt"、".mb" 文件
- 需要创建/编译/调试 MoonBit 项目
```

❌ **错误**:
```markdown
## 激活条件
当你觉得合适的时候激活本配置
```

**关键点**:
- 使用正向列表（什么时候激活）
- 提供具体的触发词或场景示例
- 同时说明"不激活"的场景（边界）

---

### SP-3: 意图路由表是核心章节

这是整个 AGENTS.md **最重要**的章节，必须精心设计。

**设计原则**:
1. 覆盖 80-90% 的常见用户请求
2. 每行都是"说 X → 做 Y"的映射
3. 按使用频率降序排列
4. 包含优先级标记（P0/P1/P2）

**标准格式**:
```markdown
| 用户意图 | 触发示例 | 执行动作 | 优先级 |
|---------|---------|---------|--------|
| 创建项目 | "新建"/"init" | → 执行 create-project 流程 | P0 |
| 编写代码 | "实现"/"写一个" | → 进入 write-code 模式 | P0 |
| 调试错误 | "报错"/"error" | → 执行 debug-flow | P0 |
```

**高级技巧**:
- 分组显示（P0 核心任务 / P1 进阶任务 / P2 专家任务）
- 添加条件判断列（如"需要提供错误信息"）
- 链接到详细的子技能文档

---

### SP-4: 章节数量控制在合理范围

避免章节过多导致信息过载。

| 类型 | 推荐章节数 | 最大值 |
|------|-----------|--------|
| Type A (最小) | 4-6 个 | 8 |
| Type B (标准) | 6-10 个 | 12 |
| Type C (框架) | 8-12 个 | 15 |
| Type D (领域) | 10-15 个 | 18 |
| Type E (系统) | 12-18 个 | 20 |

**如果超过最大值**:
- 合并相关章节
- 将低频内容移入 references/
- 提取为子技能（skills/）

---

### SP-5: 层级深度 ≤ 4 层

Markdown 标题层级不应过深。

```
✅ 推荐:
# Title (H1)
## Chapter (H2)
### Section (H3)
#### Subsection (H4)

❌ 避免:
# Title
## Chapter
### Section
#### Subsection
##### Detail (H5)
###### Sub-detail (H6)
```

**原因**:
- 过深的层级增加认知负荷
- Agent 解析时可能丢失深层信息
- 移动端显示效果差

---

### SP-6: 高频内容前置

按照"使用频率 × 重要程度"降序排列章节。

**标准顺序**:
1. 前言区 ← 解析入口
2. 身份与角色 ← WHO
3. 触发条件 ← WHEN
4. **意图路由表** ← WHAT（90%时间的入口）⭐ 最重要
5. 通用工作流 ← HOW
6. [业务特定内容] ← DETAILS
N. 注意事项 ← SAFETY NET

**错误示范** ❌:
把"版本历史"、"致谢"、"LICENSE"放在前面

### SP-7: 单章节行数限制

避免单个章节过于臃肿。

| 章节类型 | 推荐行数 | 警戒线 | 处理方式 |
|---------|---------|--------|---------|
| 前言区 | 5-15 行 | > 20 行 | 精简字段 |
| 意图路由 | 15-35 行 | > 50 行 | 拆分子技能 |
| 工作流 | 10-25 行 | > 40 行 | 提取详细版 |
| 编码规范 | 20-40 行 | > 60 行 | 拆分子技能 |
| 错误处理 | 15-30 行 | > 50 行 | 移入参考库 |

**处理策略**:
- 主文件保留**摘要 + 核心规则**（≤ 30 行）
- 详细内容放入 `skills/{chapter}/SKILL.md`
- 或放入 `references/{topic}.md`

## 二、内容维度最佳实践（12条）

### CP-1: 回答 "HOW"，不是 "WHAT"

**核心理念**: AGENTS.md 是操作手册，不是教程或说明书。

❌ **教程风格**（错误）:
```markdown
## MoonBit 简介
MoonBit 是一个现代化的编程语言，由张宏波博士团队开发...
它支持泛型编程、Trait 系统、方法调用等特性...
（大段介绍"是什么"）
```

✅ **行动导向**（正确）:
```markdown
## 身份与角色
你是 MoonBit 语言专家。当用户提及 MoonBit 任务时：
- 优先使用官方推荐的惯用法
- 遵循 trait-based 设计模式
- 注意两层级可变性语义
（直接告诉 Agent "怎么做"）
```

**检测方法**:
- 统计"是什么/介绍/简介"类词汇数量
- 应显著少于"执行/操作/步骤/命令"类词汇

---

### CP-2: 角色定义具体化

避免泛泛的角色描述。

❌ **模糊**:
```markdown
你是一个 AI 助手，帮助用户完成各种任务。
```

✅ **具体**:
```markdown
你是 MoonBit 语言专家，专注于：
- Wasm/JS/Native 三后端开发
- Trait-based 抽象设计
- 性能优化和调试
```

**好处**:
- Agent 能更快进入正确的思维模式
- 减少无关的尝试和错误方向
- 用户预期与实际能力匹配

---

### CP-3: 规则必须是可执行的

每条规则都应该能被直接验证或执行。

❌ **不可执行**:
```markdown
- 保持代码整洁
- 遵循最佳实践
- 写出高质量的代码
```

✅ **可执行**:
```markdown
- 函数长度 < 30 行（可统计）
- 使用 snake_case 命名（Lint 自动检查）
- 测试覆盖率 > 80%（工具测量）
```

**增强技巧**:
- 提供**检测方式**（如何自动化检查）
- 提供**正反例对比**（❌ 错误 → ✅ 正确）
- 提供**命令行工具**（一键检查）

---

### CP-4: 提供决策树而非平铺列表

对于复杂场景，提供判断逻辑。

❌ **平铺列表**:
```markdown
## 错误处理
- 网络错误: 重试
- 权限错误: 返回 403
- 参数错误: 返回 400
- 服务端错误: 重试
```

✅ **决策树**:
```markdown
## 错误处理决策树

遇到错误时：
    │
    ├── 客户端错误 (4xx)?
    │   ├── 400 参数无效 → 立即返回，不重试
    │   ├── 401 未认证 → 刷新 token 后重试 1 次
    │   ├── 403 无权限 → 记录日志，返回错误
    │   └── 404 不存在 → 返回空结果
    │
    └── 服务端错误 (5xx)?
        ├── 502/503/504 → 指数退避重试 (1s/2s/4s)，最多 3 次
        ├── 500 → 固定间隔重试 (1s)，最多 2 次
        └── 其他 5xx → 不重试，直接报告
```

**优势**:
- 覆盖更多边界情况
- 减少歧义和猜测
- Agent 可以直接按逻辑执行

### CP-5: 示例必须真实可用

代码示例应能在实际项目中运行（或明确标注为伪代码）。

❌ **虚假示例**:
```python
// 这是一个简单的函数
def process(data):
    # ... 处理逻辑 ...
    return result
```

✅ **真实示例**:
```python
def fetch_user(user_id: int) -> Optional[User]:
    """根据 ID 获取用户信息"""
    try:
        response = http_client.get(f"/api/users/{user_id}")
        response.raise_for_status()
        return User.model_validate(response.json())
    except HTTPError as e:
        logger.warning(f"Failed to fetch user {user_id}: {e}")
        return None
```

**要求**:
- 包含完整的导入和类型注解
- 处理边界情况和错误
- 符合项目的编码规范

### CP-7: 处理不确定性

告诉 Agent 在不确定时该怎么做。

```markdown
## 不确定性处理

当遇到以下情况时：
- **需求模糊**: 不要猜测，主动提问澄清
- **多种方案**: 列出 2-3 个选项及优劣，让用户选择
- **缺少上下文**: 先读取相关文件（README/代码），再回答
- **超出范围**: 明确告知不在能力范围内，建议替代资源
```

**为什么重要**:
- 减少 Agent 的盲目猜测
- 提高交互质量
- 避免浪费时间在错误方向

### CP-10: 使用表格呈现结构化信息

对于规则、选项、对比等内容，优先使用表格。

❌ **段落式**:
```
关于命名规范，我们应该注意以下几点：
第一，函数和变量名应该使用 snake_case 格式，
也就是全小写字母加上下划线连接的方式。
第二，类型名应该使用 CamelCase 格式，
也就是首字母大写的驼峰命名法。
第三，常量名应该使用 UPPER_SNAKE_CASE 格式，
也就是全大写字母加上下划线...
```

✅ **表格式**:
```markdown
## 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 函数/变量 | snake_case | `fetch_user`, `is_valid` |
| 类型/接口 | CamelCase | `UserService`, `HttpClient` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY`, `DEFAULT_TIMEOUT` |
| 私有成员 | _leading_underscore | `_internal_state` |
```

**优势**:
- 扫描速度提升 5x
- 信息密度提升 3x
- 易于快速查找和引用

## 三、格式维度最佳实践（6条）

### FP-1: 统一的 Markdown 规范

遵循一致的格式风格：

**标题规范**:
- H1: 仅用于文件标题（1个）
- H2: 主要章节（3-10个）
- H3: 章节内分区
- H4: 详细子项（慎用）

**列表规范**:
- 无序列项用 `-`（不用 `*` 或 `+`）
- 有序列项用于步骤流程
- 嵌套层级 ≤ 3 层

**强调规范**:
- 关键术语: **粗体**
- 警告事项: ⚠️ Emoji + 粗体
- 代码/命令: `反引号`
- 外部链接: [文本](URL)

---

### FP-2: 合理使用 Emoji 和图标

Emoji 可以提升可读性，但不要过度使用。

**推荐用法**:
- 章节标题: `## 🚀 创建项目`（帮助视觉识别）
- 状态标识: ✅/❌/⚠️（快速判断）
- 优先级: P0/P1/P2（分级展示）
- 流程箭头: → / ↓ / ← （表示流向）

**避免用法**:
- 连续多个 Emoji（如 `🎉🎊🥳`）
- 装饰性 Emoji（不影响理解的）
- 可能渲染不一致的特殊字符

---

### FP-3: 代码块标注语言

所有代码块都应指定语言类型，启用语法高亮。

```markdown
\`\`\`python
def hello():
    print("Hello, World!")
\`\`\`

\`\`\`bash
echo "Hello"
\`\`\`

\`\`\`yaml
name: my-skill
version: v1.0.0
\`\`\`
```

**好处**:
- 编辑器语法高亮，提升可读性
- 部分工具可以执行代码块（如 Jupyter）
- 明确区分伪代码和实际代码

---

### FP-4: 控制行宽和行长

保持适度的行宽，避免过长或过短。

**推荐参数**:
- 目标行宽: 80-100 字符
- 最大行宽: 120 字符（允许例外）
- 表格单元格: 尽量 < 40 字符

**处理超长行**:
- URL: 使用短链接或换行
- 长命令: 用 `\` 续行
- 长路径: 使用变量或相对路径

---

### FP-5: 链接有效性保证

确保所有内部和外部链接有效。

**内部链接**（相对路径）:
```markdown
[子技能](skills/coding-standards/SKILL.md)  ✅
[参考文档](references/error-codes.md)          ✅
```

**外部链接**（绝对 URL）:
```markdown
[官方文档](https://moonbitlang.com/docs)        ✅
[Issue #123](https://github.com/.../issues/123) ✅
```

**检查命令**:
```bash
# 检查内部链接
grep -oP '\([^\)]+\.md\)' AGENTS.md | while read link; do
  path=$(echo "$link" | sed 's/[()]//g')
  [ -f "$path" ] && echo "✅ $link" || echo "❌ 断链: $link"
done
```

建立维护机制，避免内容腐化。

**审查频率**:
- **活跃开发期**: 每月 1 次

每次修改都应记录在版本历史中，便于追溯。
