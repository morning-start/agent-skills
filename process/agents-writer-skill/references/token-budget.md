# Token 预算与效率策略

> 本文档详细说明 AGENTS.md 的 Token 消耗模型、预算分配策略和优化技巧，帮助在信息完整性和加载效率之间找到最佳平衡点。

## 为什么 Token 效率很重要

### Token 的成本

```
┌─────────────────────────────────────────────────────┐
│              Token 消耗的影响                         │
├─────────────────────────────────────────────────────┤
│                                                      │
│  💰 经济成本                                         │
│  - API 调用费用（按 Token 计费）                      │
│  - 大文件 = 每次对话都多花钱                          │
│                                                      │
│  ⏱️ 时间成本                                         │
│  - 解析时间增加（特别是首次加载）                     │
│  - Agent 响应延迟                                     │
│  - 用户体验下降                                      │
│                                                      │
│  🧠 认知负荷                                         │

│  - 决策质量下降                                      │

│  - 大文件占用宝贵的上下文空间                       │

│                                                      │
└─────────────────────────────────────────────────────┘
```

### 真实数据参考

| 文件大小 | 估算 Token 数 | 单次对话成本增量* | 月成本增量（100次/月）* |
|---------|-------------|------------------|---------------------|
| 50 行 (~1.5KB) | ~150 tokens | ~$0.002 | ~$0.20 |
| 150 行 (~4.5KB) | ~450 tokens | ~$0.007 | ~$0.70 |
| 300 行 (~9KB) | ~900 tokens | ~$0.013 | ~$1.30 |
| 500 行 (~15KB) | ~1500 tokens | ~$0.022 | ~$2.20 |
| 800 行 (~24KB) | ~2400 tokens | ~$0.036 | ~$3.60 |

**结论**: 一个 500 行的 AGENTS.md 每月可能产生 **$2+** 的额外成本（仅文件加载），还不包括因效率低下导致的额外交互轮次。

## Token 估算模型

### 粗估公式

```
Token 数 ≈ 行数 × 平均每行 Token 数

其中：
- 英文内容: ~3-4 tokens/行
- 中文内容: ~4-5 tokens/行（中文 Token 化效率较低）
- 混合内容: ~3.5-4.5 tokens/行
- 代码块: ~2-3 tokens/行（代码 Token 化效率高）
- 表格: ~2-3 tokens/行（结构化数据效率高）
```

### 精确估算方法

```bash
#!/bin/bash
# token-estimate.sh: AGENTS.md Token 消耗精确估算

FILE="${1:-AGENTS.md}"

echo "=== Token 消耗分析 ==="
echo "文件: ${FILE}"
echo ""

# 基础统计
total_lines=$(wc -l < "${FILE}")
total_chars=$(wc -m < "${FILE}")
total_bytes=$(wc -c < "${FILE}")

echo "基础统计:"
echo "  总行数: ${total_lines}"
echo "  总字符数: ${total_chars}"
echo "  总字节: ${total_bytes}"
echo ""

# 分类统计
chinese_chars=$(grep -oP '[\x{4e00}-\x{9fff}]' "${FILE}" 2>/dev/null | wc -l)
english_words=$(grep -oP '[a-zA-Z]+' "${FILE}" | wc -l)
code_blocks=$(grep -c '```' "${FILE}")
table_rows=$(grep -c '|.*|.*|' "${FILE}")
empty_lines=$(grep -c '^[[:space:]]*$' "${FILE}")

echo "内容分析:"
echo "  中文字符: ${chinese_chars}"
echo "  英文单词: ${english_words}"
echo "  代码块数: $((code_blocks / 2))"
echo "  表格行数: ${table_rows}"
echo "  空行数: ${empty_lines}"
echo ""

# Token 估算（简化模型）
# 基础：每行平均 3.5 tokens（混合内容）
base_tokens=$((total_lines * 35 / 10))

# 调整因子
chinese_penalty=$((chinese_chars / 5))      # 中文每5字符多1token
code_bonus=$(( (code_blocks / 2) * -20 ))   # 代码块每块少20token
table_bonus=$((table_rows * -2))             # 表格每行少2token
empty_penalty=$((empty_lines * 2))           # 空行每行2token（浪费）

estimated_tokens=$((base_tokens + chinese_penalty + code_bonus + table_bonus + empty_penalty))

if [ $estimated_tokens -lt 0 ]; then
  estimated_tokens=100
fi

echo "Token 估算:"
echo "  基础估算: ~${base_tokens} tokens"
echo "  中文调整: +${chinese_penalty} tokens"
echo "  代码调整: ${code_bonus} tokens"
echo "  表格调整: ${table_bonus} tokens"
echo "  空行调整: +${empty_penalty} tokens"
echo "  ─────────────────────"
echo "  **最终估算: ~${estimated_tokens} tokens**"
echo ""

# 效率评级
tokens_per_line=$((estimated_tokens * 10 / total_lines))
echo "效率指标:"
echo "  平均 Token/行: ${tokens_per_line}.${((estimated_tokens * 10 % total_lines))}"

if [ $estimated_tokens -lt 300 ]; then
  echo "  效率评级: ✅ **优秀** (< 300 tokens)"
  echo "  建议: 当前大小理想，无需优化"
elif [ $estimated_tokens -lt 600 ]; then
  echo "  效率评级: ✅ **良好** (300-600 tokens)"
  echo "  建议: 可接受范围，如需优化见下方策略"
elif [ $estimated_tokens -lt 1000 ]; then
  echo "  效率评级: ⚠️ **一般** (600-1000 tokens)"
  echo "  建议: 推荐进行 Token 优化（见 Phase 1-3）"
else
  echo "  效率评级: ❌ **较低** (> 1000 tokens)"
  echo "  建议: **强烈建议**立即优化（可能影响性能）"
fi

echo ""
echo "成本估算（GPT-4o 级别）:"
cost_per_call=$(echo "scale=4; ${estimated_tokens} * 0.000015" | bc)
cost_monthly=$(echo "scale=2; ${cost_per_call} * 100" | bc)
echo "  单次加载成本: ~\$${cost_per_call}"
echo "  月成本（100次/月）: ~\$${cost_monthly}"
```

**使用方法**:
```bash
chmod +x token-estimate.sh
./token-estimate.sh AGENTS.md
```

## 各类型的 Token 预算推荐

### 预算分配原则

```
总 Token 预算 = f(类型, 团队规模, 使用频率)

基础公式:
  Type A: 100-250 tokens (30-80行)
  Type B: 250-600 tokens (80-200行)
  Type C: 600-1200 tokens (200-400行)
  Type D: 1200-1800 tokens (400-600行)
  Type E: 1800-3000 tokens (600-1000行)
```

### 详细预算表

| 类型 | 最小 Token | 推荐 Token | 最大 Token | 对应行数 | 占比说明 |
|------|-----------|-----------|-----------|---------|---------|
| **A** | 100 | 180 | 250 | 30-80 | 仅核心规则 |
| **B** | 250 | 400 | 600 | 80-200 | 路由+规范+命令 |
| **C** | 600 | 900 | 1200 | 200-400 | +子技能索引+知识库指针 |
| **D** | 1200 | 1500 | 1800 | 400-600 | +领域知识摘要 |
| **E** | 1800 | 2400 | 3000 | 600-1000 | +合规/安全/运维 |

### Token 分配策略（以 Type C 为例）

假设 **900 tokens** 预算，如何分配？

```
┌─────────────────────────────────────────────┐
│        Type C Token 预算分配 (900 total)     │
├──────────────────┬──────────┬────────────────┤
│ 组件              │ Token数  │ 占比           │
├──────────────────┼──────────┼────────────────┤
│ 前言区 (YAML)    │ 30       │ 3.3%           │
│ 身份与角色       │ 40       │ 4.4%           │
│ 触发条件         │ 25       │ 2.8%           │
│ 意图路由表       │ 80       │ 8.9%  ← 核心!  │
│ 通用工作流       │ 60       │ 6.7%           │
│ 编码规范         │ 100      │ 11.1%          │
│ 常用命令         │ 80       │ 8.9%           │
│ 错误处理         │ 70       │ 7.8%           │
│ 质量门禁         │ 45       │ 5.0%           │
│ 注意事项         │ 40       │ 4.4%           │
│ 子技能索引       │ 60       │ 6.7%           │
│ 知识库指针       │ 50       │ 5.6%           │
│ 版本历史         │ 30       │ 3.3%           │
│ 格式开销(空行等) │ 100      │ 11.1%          │
│ 缓冲(预留)       │ 100      │ 11.1%          │
├──────────────────┼──────────┼────────────────┤
│ 合计             │ 900      │ 100%           │
└──────────────────┴──────────┴────────────────┘
```

**关键洞察**:
1. **意图路由**仅占 8.9% Token，但贡献 **40%+** 的价值 → **高 ROI**
2. **格式开销**（空行、Markdown语法）占 11.1% → **低 ROI**，可优化
3. **编码规范**占 11.1%，但使用频繁 → **中等 ROI**

## Token 优化策略

### 策略 1: 删除冗余（最安全，收益 10-20%）

**目标**: 移除不增加价值的内容

| 冗余类型 | 示例 | Token 节省 | 风险 |
|---------|------|-----------|------|
| 过度解释 | "为什么需要 X"的长篇论述 | -50~150 | 极低 |
| 重复内容 | 同一信息出现 2+ 次 | -30~80 | 低 |
| 空行过多 | 连续 > 2 个空行 | -20~50 | 极低 |
| 废话开头 | "首先"、"然后"、"此外" | -20~40 | 低 |
| 过时信息 | 已废弃版本的说明 | -30~100 | 低 |

**操作**:
```bash
# 快速删除多余空行
sed -i '/^$/N;/^\n$/d' AGENTS.md

# 统计可删除的过渡词
grep -ioE "(此外|另外|值得注意的是|总的来说)" AGENTS.md | wc -l
```

---

### 策略 2: 表达压缩（收益 15-25%，需审核）

**目标**: 用更少的 Token 传达相同信息

| 压缩技巧 | 前 | 后 | 节省率 |
|---------|-----|-----|--------|
| 段落→要点 | 200字符×5行 | 150字符×3列表 | -55% |
| 叙述→表格 | 300字符段落 | 200字符表格 | -33% |
| 完整句→短语 | "你应该..." | "做..." | -20% |
| 长名→短名 | "请执行以下操作" | "操作:" | -60% |

**示例**:

❌ **压缩前** (约 250 tokens):
```markdown
关于代码格式化的重要性，我想在这里特别强调一下。
在我们的项目中，保持一致的代码格式是非常重要的，
因为它不仅能够提高代码的可读性，还能够减少代码审查
时的不必要的讨论。因此我们强烈建议所有的团队成员
在编写代码的时候都要遵循统一的格式化规范...
```

✅ **压缩后** (约 80 tokens, -68%):
```markdown
## 格式化要求
- **强制**: 保存时自动格式化 (EditorConfig + Prettier)
- **检查**: CI 拒绝未格式化的 PR
- **例外**: 需在 PR Description 中说明原因
```

---

### 策略 3: 结构重组（收益 20-40%，需测试）

**目标**: 将低频内容移出主文件

**判断标准**（满足任一即考虑提取）：
- 单章节 > 50 行
- 内容查阅频率 < 30%
- 不是每次对话都需要的信息

**提取方案对比**:

| 目标位置 | 适用内容 | Token 影响 | 访问成本 |
|---------|---------|-----------|---------|
| `skills/X/SKILL.md` | 可独立使用的子流程 | 主文件 -30~50 | 需显式调用 |
| `references/X.md` | 详细参考资料 | 主文件 -20~40 | 按需读取 |
| 外部链接 | 第三方文档 | 主文件 -10~20 | 网络请求 |

**推荐优先级**: skills/ > references/ > 外部链接

---

### 策略 4: 格式优化（收益 5-15%，低风险）

**目标**: 减少 Markdown 格式本身的开销

| 优化项 | 操作 | Token 节省 |
|--------|------|-----------|
| 减少 H4/H5 层级 | 合并到 H3 | -10~30 |
| 精简表格列 | 只保留必要列 | -20~50 |
| 减少代码块 | 用行内代码替代短示例 | -15~40 |
| Emoji 精简 | 仅保留关键位置的 | -10~20 |
| 删除注释性文字 | (如本句) | -20~60 |

**注意**: 不要为了节省 Token 牺牲可读性。格式的目的是**提升扫描效率**，这本身就是一种 Token 节省（Agent 找信息更快 = 更少轮次）。

## 高效写作模式

### 模式 1: 表格优于叙述（永远成立）

**Rule**: 只要能用表格呈现的结构化信息，**绝不**用段落。

**数据支撑**:
- 表格的信息密度: **3-5x** 段落
- 扫描速度: **5x** 段落
- Token 效率: **1.5-2x** 段落（相同信息量）

**适用场景**:
✅ 规则列表、选项对比、配置参数、状态码、命令速查

❌ 不适用: 叙述性流程、背景介绍、决策逻辑（用 ASCII art）

---

### 模式 2: 要点列表优于长段落（90% 成立）

**Rule**: 超过 3 行的段落，考虑拆为要点列表。

**阈值**:
- < 3 行: 保持段落
- 3-6 行: 可选拆分
- > 6 行: **必须拆分**

---

### 模式 3: 链接优于内联（按需成立）

**Rule**: 低频详细信息用链接指向，不在主文件展开。

```markdown
❌ 内联展开（主文件膨胀 +150 tokens）:
## 错误码完整列表
E001: 类型不匹配 - 当赋值表达式左右两侧的类型不一致时触发...
E02: 未定义变量 - 当使用了未声明的变量时触发...
（20个错误码，每个30字 = 600 tokens）

✅ 链接引用（主文件仅 +30 tokens）:
## 常见错误
→ 详见 [错误码完整表](references/error-codes.md) (TOP 5 见下)

| 错误 | 快速解决 |
|------|---------|
| E001 | 检查类型注解 |
| E02 | 检查变量声明 |
... (仅列出5个最高频的)
```

**判断标准**:
- 用户**每次**都需要? → 内联
- 用户**有时**需要? → 内联 TOP N + 链接完整版
- 用户**很少**需要? → 仅链接

## Token 预算实战案例

### 案例: moonbit-skills v9 → v10 的 Token 优化

**v9 数据**:
- 总行数: 606 行
- 估算 Token: ~1800
- 问题: 教程风格，大量背景介绍

**v10 数据**:
- 总行数: 156 行
- 估算 Token: ~470
- 优化: **-74% Token**

**优化手段**:

| 手段 | 节省 Token | 占比 |
|------|-----------|------|
| 删除教程化内容 | -800 | 57% |
| 提取至 references/ | -350 | 25% |
| 表达压缩 | -150 | 11% |
| 意图路由替代目录 | -30 | 2% |
| 其他（空行/格式） | -70 | 5% |
| **总计** | **-1400** | **100%** |

**质量影响**: 
- 功能完整性: **无损**（所有知识移入 references/）
- 使用效率: **提升**（路由表比目录更高效）
- 维护成本: **降低**（主文件更易修改）

## 监控与维护

### 定期 Token 审计

**推荐频率**: 每季度一次（或重大变更后）

**审计清单**:
```bash
# 1. 统计当前 Token 消耗
./token-estimate.sh AGENTS.md

# 2. 对比上次审计结果（如有记录）
# 3. 识别异常增长（> 20% 需调查原因）
# 4. 检查是否有新注入的冗余内容
# 5. 评估是否需要触发优化流程
```

### Token 增长预警阈值

| 增长幅度 | 触发动作 |
|---------|---------|
| < 10% | 正常波动，无需处理 |
| 10-25% | 记录日志，下次审计关注 |
| 25-50% | ⚠️ 调查原因，考虑轻量优化 |
| > 50% | 🔴 **立即审查**，可能需要重构 |

### 自动化监控（进阶）

```yaml
# .github/workflows/token-audit.yml（示例）
name: Token Audit

on:
  schedule:
    - cron: '0 0 1 * *'  # 每月1号
  workflow_dispatch:

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Estimate Tokens
        run: |
          chmod +x scripts/token-estimate.sh
          ./scripts/token-estimate.sh AGENTS.md > token-report.txt
          
      - name: Check Threshold
        run: |
          TOKENS=$(grep "最终估算" token-report.txt | grep -oP '\d+')
          if [ "$TOKENS" -gt 1000 ]; then
            echo "::warning::AGENTS.md Token count (${TOKENS}) exceeds threshold (1000)"
            exit 1
          fi
```

## 总结：Token 效率黄金法则

### 🥇 法则 1: 价值密度优先
> 每 Token 都应该承载**可执行的信息**，而非背景知识。

### 🥈 法则 2: 渐进式披露
> 20% 的内容解决 80% 的问题（放在主文件），其余 20% 的问题由 80% 的内容解决（放在 references/）。

### 🥉 法则 3: 结构化永胜
> 表格 > 列表 > 段落（对于规则/选项/对比类内容）。

### 🏅 法则 4: 测量再优化
> 先测量（token-estimate.sh），再优化，最后验证。不要凭感觉。

### 🏅 法则 5: 定期审计
> Token 膨胀是自然趋势，每季度审查一次，保持健康。
