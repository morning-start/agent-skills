# 安全检查清单

> 用于阶段 1（识别安全需求）和阶段 4（验证安全门禁）的完整安全红线清单。

---

## 安全等级判定

| 等级 | 判断标准 | AGENTS.md 安全性要求 |
|------|---------|-------------------|
| 🔴 **关键** | 涉及用户隐私数据(PII)、支付、金融交易、医疗信息 | 必须包含详细的安全章节 + 多层门禁 |
| 🟠 **高** | 涉及生产环境操作、API 密钥、认证系统 | 必须包含红线章节 + 至少 1 个硬门禁 |
| 🟡 **中** | 涉及网络访问、文件系统写操作、外部服务调用 | 建议包含安全规则 |
| 🟢 **低** | 纯本地脚本、无网络、无敏感数据 | 基础安全提示即可 |

---

## 安全红线清单

### 1. 凭证管理

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| API 密钥硬编码 | 永远不要将 API 密钥写入代码，使用环境变量或密钥管理服务 | 🔴 HARD |
| 提交 .env 文件 | .env 必须加入 .gitignore，检测到 .env 文件时阻止提交 | 🔴 HARD |
| 密码/令牌输出 | 不要在生产日志中输出密码、令牌或密钥 | 🟠 HARD |
| 配置文件泄露 | 包含敏感信息的配置文件（如 production.yaml）不要公开 | 🟠 HARD |
| 临时凭证 | 临时生成的凭证在会话结束后必须清理或撤销 | 🟡 SOFT |

### 2. 数据安全

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| 用户数据泄露 | 不要将用户数据写入日志、控制台或非安全存储 | 🔴 HARD |
| PII 处理 | 处理个人身份信息时必须脱敏或匿名化 | 🔴 HARD |
| 数据备份覆盖 | 不要覆盖用户数据，除非明确指定且数据已备份 | 🟠 HARD |
| 数据批量导出 | 大规模数据导出需要用户确认 | 🟠 SOFT |
| 测试数据 | 不要使用生产数据的完整副本作为测试数据 | 🟡 SOFT |

### 3. 文件系统安全

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| 危险命令 | 永远不要执行 `rm -rf /`、`dd`、`fdisk` 等破坏性命令 | 🔴 HARD |
| 生产文件修改 | 修改生产环境的配置文件需要用户确认 | 🔴 HARD |
| 权限变更 | 不要修改文件权限（chmod）或所有者（chown） | 🟠 HARD |
| 大量文件删除 | 批量删除操作前必须预览删除列表并确认 | 🟠 SOFT |
| 符号链接 | 不要跟踪或创建指向系统目录的符号链接 | 🟠 SOFT |

### 4. 网络与外部服务

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| 未知 URL | 不要向用户未指定的 URL 发送请求 | 🟠 SOFT |
| 外部依赖安装 | 安装外部包或工具需要用户确认 | 🟠 SOFT |
| 付费 API 调用 | 调用付费 API 前必须告知用户可能产生的费用 | 🟠 HARD |
| 内网扫描 | 不要扫描内部网络或发现服务 | 🟠 SOFT |
| 代码执行 | 不要 eval 或 exec 用户未提供的代码字符串 | 🟠 HARD |

### 5. 代码变更安全

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| 直接推送主分支 | 不要直接推送到 main/master，使用分支+PR | 🟠 HARD |
| CI/CD 修改 | 修改 CI/CD 配置文件（.github/workflows/）需要用户确认 | 🟠 HARD |
| 第三方依赖升级 | 升级第三方库的主要版本需要验证兼容性 | 🟡 SOFT |
| 数据库迁移 | 生产数据库的 DDL 操作需要用户手动执行 | 🔴 HARD |
| 安全补丁 | 已知漏洞的依赖更新应标记为高优先级 | 🟡 SOFT |

### 6. 非功能约束

| 风险 | 规则 | 门禁级别 |
|------|------|---------|
| 无限循环 | 确保所有循环有终止条件，避免资源耗尽 | 🟠 SOFT |
| 大文件提交 | 提交 > 100MB 的文件前警告用户 | 🟡 SOFT |
| 并发操作 | 并发修改共享资源时需要锁或事务 | 🟡 SOFT |
| 日志过度 | 不要记录敏感信息到日志文件 | 🟠 SOFT |
| 第三方组件许可 | 引入新依赖时检查其许可证是否兼容 | 🟡 SOFT |

## 项目安全检查速查表

在阶段 1 分析时，使用此表快速评估项目安全等级：

```bash
# 安全检查速查
echo "=== 安全检查速查 ==="

# 检查 .env
if [ -f ".env" ]; then
  echo "⚠️ .env 文件存在"
fi

# 检查 .gitignore
if [ -f ".gitignore" ]; then
  if grep -q ".env" .gitignore; then
    echo "✅ .env 在 .gitignore 中"
  else
    echo "⚠️ .env 不在 .gitignore 中"
  fi
fi

# 检查 API 密钥模式
key_count=$(grep -rE "(API_KEY|SECRET|PASSWORD|TOKEN)" --include="*.{ts,js,py,rs,go}" . 2>/dev/null | grep -v "node_modules" | grep -v ".env" | wc -l)
if [ $key_count -gt 0 ]; then
  echo "⚠️ 发现 $key_count 处可能的密钥硬编码"
fi

# 检查生产环境配置
if [ -d "deploy" ] || [ -f "docker-compose.yml" ] || [ -f "Dockerfile" ]; then
  echo "ℹ️ 存在部署配置（deploy/docker-compose/Dockerfile）"
fi

# 检查 CI/CD
if [ -d ".github/workflows" ]; then
  echo "ℹ️ 存在 GitHub Actions 配置"
fi

# 检查测试
test_count=$(find . -name "test*" -o -name "*test*" -o -name "spec*" 2>/dev/null | grep -v "node_modules" | wc -l)
if [ $test_count -gt 0 ]; then
  echo "✅ 存在 $test_count 个测试文件"
else
  echo "⚠️ 未发现测试文件"
fi

echo "=== 速查完成 ==="
```
