## 概览

决策追踪是将讨论结果系统化记录的关键，确保：
- 决策有据可查
- 讨论结论不丢失
- 后续可追溯决策理由
- 避免重复讨论

## 讨论流程

### 讨论前准备

1. **收集问题**：将问题按类别组织
2. **确定优先级**：P0 > P1 > P2
3. **准备材料**：竞品分析数据、技术对比表格

### 讨论中记录

```
讨论议题：D-001 MVP 插件类型

观点 A：先做 QuickJS
├── 理由：开发快、风险低
├── 风险：功能受限
└── 支持者：开发者

观点 B：先做 Webview
├── 理由：支持复杂 UI
├── 风险：开发慢、内存大
└── 支持者：无

结论：选择 QuickJS
理由：MVP 需要快速验证
```

### 讨论后归档

1. **更新决策矩阵**：将结论写入
2. **更新相关文档**：同步到 SRS.md
3. **通知相关方**：告知决策结果

## 文档结构

```

├── 00-summary.md          # 决策汇总（所有决策索引）
├── 01-project-management.md   # 项目管理问题
├── 02-technical-architecture.md  # 技术架构问题
```

## 更新机制

### 决策变更流程

```
发现新信息
    │
    ▼
评估影响范围
    │
    ├── 轻微变化 → 直接更新
    │
    ├── 重大变化 → 重新讨论
    │
    └── 无变化 → 忽略
```

### 版本管理

```markdown
## 变更日志

### v1.1.0 - 2026-04-03
- 新增决策 D-005：插件签名方案
- 更新决策 D-003：分发机制从 npm 改为 GitHub Release
- 原因：简化分发流程
```

### 同步到 SRS

每次决策确定后，应同步更新 SRS.md：

```markdown
## SRS.md 更新清单

- [x] 1.1 MVP 边界定义
- [x] 2.1 插件 API 兼容性
- [x] 2.2 QuickJS 集成细节
- ...

更新时间：2026-04-03
```
