# ⑥ 迭代优化与维护

## 任务目标

对已有 AGENTS.md 进行**持续性优化**，包括 Token 效率提升、内容精简、结构重组和版本演进策略，确保文件随项目成长而持续增值，避免膨胀和腐化。

## 触发条件

当用户说以下任一话术时激活：
- "优化我的AGENTS.md"
- "减少Token占用"
- "太长了，精简一下"
- "重构结构"
- "迭代改进"
- "维护更新"

## 优化类型分类

```
需要什么类型的优化？
    │
    ├── 📉 Token优化 → 减少体积，提高加载速度
    │   触发词: "太长"/"Token"/"精简"/"压缩"
    │
    ├── 🔄 结构重组 → 改善组织，提升可读性
    │   触发词: "重构"/"重组"/"结构调整"
    │
    ├── ✨ 内容增强 → 补充缺失，提升质量
    │   触发词: "增强"/"补充"/"完善"
    │
    └── 📈 版本演进 → 适配变化，跟随项目成长
        触发词: "更新"/"升级"/"演进"/"v2"
```

#### 1.2 优化策略矩阵

| 策略 | 适用场景 | 预期效果 | 风险 |
|------|---------|---------|------|
| **删除冗余** | 重复内容、废话、过时信息 | -10~30% | 低 |
| **压缩表达** | 啰嗦段落、过度解释 | -15~25% | 中 |
| **提取子技能** | 某章节 > 50 行 | -20~40% | 中 |
| **移入参考库** | 低频查阅的详细内容 | -30~50% | 低 |
| **表格化** | 规则列表、选项对比 | -10~20% | 低 |
| **合并章节** | 相关的小章节 | -5~15% | 中 |

#### 1.3 具体优化技术

##### 技术 1: 删除冗余（安全，优先执行）

**目标**: 移除不增加价值的内容

```markdown
❌ 删除前:
## 项目简介
本项目是一个基于 MoonBit 语言开发的技能库系统，
旨在为 AI Agent 提供高质量的编程辅助能力。
MoonBit 是一个新兴的现代化编程语言，
支持 Wasm、JS 和 Native 三种后端编译 target...
（大段背景介绍，100+行）

✅ 删除后:
（完全删除此章节，或压缩为 1-2 行放入身份章节）
```

**检查清单**：
- [ ] 大段的"什么是 X"介绍（Agent 通常已经知道）
- [ ] 重复出现的信息（同一件事说了 2 遍）
- [ ] 过时的版本说明或已废弃的 API 提及
- [ ] 过度的"为什么"解释（保留结论即可）
- [ ] 空行过多（连续 > 2 个空行合并为 1 个）

##### 技术 2: 压缩表达（需谨慎，保持可读性）

**目标**: 用更少的字传达相同的信息

```markdown
❌ 优化前:
## 编写测试的重要性
编写测试对于保证软件质量非常重要。
测试可以帮助我们在修改代码的时候
确保不会引入新的 bug，同时也可以
作为文档帮助其他开发者理解代码
的预期行为。因此我们应该养成...
（80字符 × 5行 = 400字符）

✅ 优化后:
## 测试策略
- 目的: 回归防护 + 活文档
- 覆盖要求: 核心路径 100%, 全局 > 80%
- 工具: {test_framework}
（~80字符 × 3行 = 240字符，节省 40%）
```

**压缩技巧**：
- **长句 → 要点列表**: 将段落拆为 3-5 个要点
- **主动语态**: "你应该做 X" → "做 X"
- **删除过渡词**: "因此"、"此外"、"值得注意的是"
- **用符号代替文字**: "第一步、第二步、第三步" → "①②③"
- **表格替代叙述**: 规则类内容必须表格化

##### 技术 3: 提取子技能（中等复杂度）

**目标**: 将大型章节移入 skills/ 目录

**判断标准**（满足任一即考虑提取）：
- 单章节 > 50 行
- 章节内容相对独立（可单独使用）
- 该内容的查阅频率 < 30%

**操作流程**：

```bash
# 1. 创建子技能目录
mkdir -p skills/{chapter-slug}

# 2. 提取内容到新文件
cat > skills/{chapter-slug}/SKILL.md << 'EOF'
---
name: {chapter-slug}
version: v1.0.0
author: {original-author}
description: {从原章节提取的描述}
tags: [{related-tags}]
dependency:
  parent: agents-writer
---

# {原章节标题}

{原章节完整内容}
EOF

# 3. 在主文件中替换为链接
# 原:
## {章节标题}
{大量内容...}

# 后:
## {章节标题}
→ 详见 [{章节名称}](skills/{chapter-slug}/SKILL.md)
```

**示例**：

```markdown
❌ 提取前（主文件 450 行）:
## 编码规范
### 必须遵守的规则
（30行的规则表格）

### 推荐惯例
（25行的惯例列表）

### 格式化要求
（15行的格式说明）

### 命名规范详解
（40行的命名规则）

### 代码示例
（50行的示例代码）
...总计: 160+ 行

✅ 提取后（主文件 320 行，-130 行）:
## 编码规范
→ 详见 [编码规范完整版](skills/coding-standards/SKILL.md)

### 核心规则速查（保留在主文件，5行）
| # | 规则 | 一句话 |
|---|------|--------|
| R1 | snake_case | 函数/变量命名 |
| R2 | CamelCase | 类型命名 |
| R3 | 4空格缩进 | 统一格式 |
```

##### 技术 4: 移入参考库（低风险）

- 完整的错误码表（> 20 个）
- 扩展阅读和参考资料
- 版本历史（> 3 个版本）

**操作**：

##### 技术 5: 表格化转换（高质量）

**目标**: 将叙述性文本转换为紧凑的表格

```markdown
❌ 表格化前:
关于错误处理，我们需要注意以下几点：
第一，当遇到网络错误时，应该进行最多3次重试，
每次间隔应该采用指数退避策略，也就是1秒、
2秒、4秒这样的方式。第二，如果重试3次仍然
失败，应该返回一个明确的错误给调用者，而不
是吞掉错误。第三，对于权限不足的情况，应该
记录日志并返回 403 状态码...

（200字符，难以快速扫描）

✅ 表格化后:
## 错误处理规则

| 错误类型 | 处理策略 | 重试 | 返回值 |
|---------|---------|------|--------|
| 网络超时 | 指数退避重试 | 3次 (1s/2s/4s) | 超时错误 |
| 权限不足 | 记录日志 | 不重试 | 403 |
| 参数无效 | 立即返回 | 不重试 | 400 |
| 服务端错误 | 固定间隔重试 | 2次 (1s/1s) | 500 |

（250字符，但信息密度提升 3x，扫描速度提升 5x）
```

### Phase 2: 结构重组（改善可读性）

#### 2.1 诊断结构问题

**常见症状**：

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| 找不到需要的信息 | 章节顺序不合理 | 重新排列（高频前置） |
| 某章节特别长 | 内容堆积 | 拆分为子章节或子技能 |
| 相似内容分散 | 缺乏整合 | 合并相关章节 |
| 层级太深或太浅 | 层级不平衡 | 标准化为 3-4 层 |
| 缺少导航 | 大文件无索引 | 添加目录或快速导航 |

#### 2.2 重组策略

**策略 A: 重新排序（低风险）**

原则：**使用频率 × 重要程度**降序

```
新顺序建议:
1. 前言区 (YAML) ← 解析入口
2. 身份与角色 ← WHO
3. 触发条件 ← WHEN
4. 意图路由表 ← WHAT（90%时间的入口）
5. 通用工作流 ← HOW（主流程）
6. [核心业务...] ← DETAILS（按需深入）
   ├─ 编码规范
   ├─ 常用命令
   └─ 错误处理
N. 注意事项 ← SAFETY NET（最后保障）
```

**策略 B: 章节拆分（中风险）**

标准：某章节 > 60 行时考虑拆分

```markdown
❌ 拆分前:
## 开发指南（120行）
### 环境搭建（30行）
### 编码规范（40行）
### 测试策略（25行）
### 部署流程（25行）

✅ 拆分后:
## 开发指南
→ 本章节包含以下子主题：
  - [环境搭建](#环境搭建) (30行)
  - [编码规范](#编码规范) (40行) ← 如仍过长，可提取为子技能
  - [测试策略](#测试策略) (25行)
  - [部署流程](#部署流程) (25行)
```

**策略 C: 层级标准化（低风险）**

推荐层级结构：

```
# Title (H1, 仅1个，文件标题)
## Chapter (H2, 主要章节，3-8个)
### Section (H3, 章节内的分区)
#### Subsection (H4, 详细子项，慎用)
##### Detail (H5, 尽可能避免)
```

**检查命令**：

```bash
# 检查层级跳跃
grep -n "^#" AGENTS.md | awk -F'#' '{print length($2)-1, $0}' | awk '$1>{NR-1}+1 && NR>1'

# 统计各层级数量
echo "H2章节数: $(grep -c '^## ' AGENTS.md)"
echo "H3节数: $(grep -c '^### ' AGENTS.md)"
echo "H4小节数: $(grep -c '^#### ' AGENTS.md)"
```

### Phase 4: 版本演进策略

#### 4.1 语义化版本规则

遵循 skill-factory 的版本规范：

| 变更类型 | 版本更新 | Commit 前缀 |
|---------|---------|------------|
| 修复错别字/格式 | patch (+0.0.1) | `fix` |
| 补充内容/新增小章节 | minor (+0.1.0) | `feat` |
| 结构重组/优化 | minor (+0.1.0) | `refactor` |
| 破坏性变更（删除核心章节） | major (+1.0.0) | `feat!` |

#### 4.2 版本更新流程

```bash
# 1. 更新前言区的 version 字段
sed -i 's/^version: v.*/version: v{NEW_VERSION}/' AGENTS.md

# 2. 更新版本历史（追加到末尾）
cat >> AGENTS.md << EOF

## 版本历史

| 版本 | 日期 | 主要变更 |
|------|------|---------|
| **v{NEW_VERSION}** | {date} | {change_summary} |
| v{OLD_VERSION} | {old_date} | {old_changes} |
EOF

# 3. Git commit
git add AGENTS.md
git commit -m "{type}(agents): {concise_message}"
```

## 常见问题

**Q: 优化到什么程度合适？**

A: 取决于项目类型：
- **个人项目**: B 级 (75-89分) 即可，不要过度优化
- **团队项目**: A 级 (90-100分)，特别是高频使用的
- **开源项目**: A 级 + 定期维护（公众形象）

**Q: 优化的频率应该是多少？**

A: 建议：
- **活跃开发期**: 每月 1 次轻量优化
- **稳定期**: 每季度 1 次审查
- **重大变更后**: 立即优化

**Q: 可以自动化优化吗？**

A: **部分可以**：
- ✅ 可自动化：删除空行、统计行数、检测反模式
- ❌ 需人工：内容质量判断、结构重组决策、优先级排序
- 💡 推荐：结合 [⑤ 质量检查](../skills/5-quality-check/SKILL.md) 的自动检测 + 人工决策

**Q: 优化后功能会受影响吗？**

A: **如果操作正确，不会**。优化原则：
1. 每次只改一处
2. 改完后立即验证（重新运行质量检查）
3. 保留备份（git commit）

## 注意事项

⚠️ **渐进式优化**: 不要试图一次性完美。分阶段执行，每个阶段都验证。

⚠️ **保留价值**: 优化的目标是**提高信噪比**，不是简单地删除内容。如果有价值但低频，移入 references/ 而非删除。

⚠️ **用户反馈驱动**: 最终的优化方向应以实际使用反馈为准，而非理论最优。

⚠️ **版本兼容性**: 如果 AGENTS.md 被多个 Agent 或工具读取，优化时要考虑向后兼容性。
