# 需求分解器 — 详细方法论

## 核心思维模型

### 1. 需求模糊度矩阵

根据两个维度判断需求的清晰程度：

| 维度 | 低 | 高 |
|------|-----|-----|
| **目标明确性** | 不知道要做什么 | 目标清晰 |
| **约束完整性** | 几乎没有限制条件 | 约束充分 |

```
                约束完整
                  高 │
                     │    ✅ 直接执行
                     │
目标明确性 ──────────┼────────────
         低 │       │
            │    ⚠️ 需拆解（最常见）
            │    🔴 需大量澄清
          低 │
```

### 2. 模糊点检测模式

以下信号提示存在需要处理的模糊点：

**词汇信号：**
- "大概"、"大约"、"可能"、"应该"、"尽量"
- "一个好的"、"高效的"、"优雅的"、"简单的"
- "等等"、"之类"、"什么的"
- "适当处理"、"合理"、"视情况"

**结构信号：**
- 只有目标，没有输入输出说明
- 有功能描述，没有非功能约束（性能/安全/兼容）
- 有正向描述，没有异常/边界情况
- 单数场景，缺少批量/并发/错误场景

**上下文信号：**
- 明确说是"作业"/"面试题"→ 故意模糊是题目设计的一部分
- 来自产品经理 → 可能隐含了业务上下文没说出来
- 来自技术文档 → 可能省略了"显而易见"的技术约定

## 各 Phase 详细操作指南

### Phase 1: RESTATE — 复述

**目的：** 建立共识基线，尽早发现理解偏差。

**技巧：**
- 用"用户故事"格式复述："作为[角色]，我希望[功能]，以便[价值]"
- 显式标注哪些是原文有的、哪些是你的推断（用 [推断] 标记）
- 如果需求超过3句话，用 bullet points 总结要点

**常见错误：**
- ❌ 在复述阶段就加入自己的"优化建议"
- ❌ 省略看似不重要的细节（往往后面证明关键）
- ✅ 忠实复述 + 显式标注不确定处

### Phase 2: EXTRACT — 结构化提取

**7维框架逐项检查清单：**

```
□ Goal:     用一句话能说清目标吗？
□ Actor:    谁触发？谁消费？有权限角色吗？
□ Input:    数据从哪来？格式？量级？更新频率？
□ Output:   输出给谁？格式？实时还是批处理？
□ Constraint: 时间？空间？技术栈？兼容性？
□ NFR:      并发？可用性？安全？可维护性？
□ Context:  为什么有这个需求？背景故事？
```

**每个维度追问句式：**
- Goal: "如果做完了，怎么判断成功了？"
- Actor: "除了使用者还有谁关心这个结果？"
- Input: "最坏情况的输入是什么样的？（最大/最小/空/非法）"
- Output: "调用方拿到结果后会立刻做什么？"
- Constraint: "有没有绝对不能用的技术/方案？"
- NFR: "同时会有多少人在用？数据会增长到多大？"
- Context: "这个需求解决什么痛点？替代方案是什么？"

### Phase 3: CLARIFY — 歧义分类

**三分类决策树：**

```
这个模糊点不确认的话：
├── 会导致选完全不同的技术路线？ → 🔴 Blocker
├── 有合理默认值且事后可调整？   → 🟡 Assumption
└── 不影响当前阶段的任何决策？     → 🟢 Defer
```

**提问技巧：**
- **封闭式优先**：先问 Yes/No 问题，降低用户回答成本
- **给选项而非填空**："你希望 A 还是 B？"比"你希望怎样？"好
- **一次问完**：把所有 🔴 问题集中列出，避免来回反复
- **附上推荐**：对 🟡 项目直接给出假设和理由，用户只需 say No

**假设声明模板：**
> **假设 [X]**：我假设 [具体内容]，因为 [理由]。
> 如果不对请纠正，否则我将按此推进。

### Phase 4: DECOMPOSE — 任务分解

**分解方法选择：**

| 方法 | 适用场景 | 示例 |
|------|----------|------|
| **流程拆解** | 有明确步骤的任务 | 编译器：词法→语法→语义→代码生成 |
| **数据流拆解** | 数据驱动的任务 | ETL：抽取→转换→加载 |
| **模块拆解** | 可并行的任务 | 微服务：用户/订单/支付 |
| **分层拆解** | 有架构层次的任务 | MVC / Clean Architecture |

**叶子节点检验标准：**
一个步骤是否足够小？问自己：
1. 能否在 30 分钟内独立完成？✅ 足够小
2. 是否不需要再问"怎么做"就能动手？✅ 足够清楚
3. 交付物是否可以用一句话描述？✅ 足够具体

**依赖关系标注规则：**
- `A → B`：B 需要 A 的产出才能开始
- `A ↔ B`：互相依赖，需合并或排序
- `A || B`：可并行，无依赖

### Phase 5: PLAN — 行动计划

**优先级排序原则（MoSCoW 变体）：**
- **P0 Must**：不做就不算完成核心需求
- **P1 Should**：不做能用但不完整
- **P2 Could**：有了更好，没有也行

**风险识别 checklist：**
- □ 技术风险：没用过的技术/库/模式？
- □ 依赖风险：第三方 API / 外部服务？
- □ 数据风险：输入可能为空/超大/格式错？
- □ 时间风险：某个子任务可能耗时远超预期？
- □ 沟通风险：还有未确认的 🔴 Blocker？

## 示例：编程作业拆解

### 原始需求
> "写一个函数，输入一些数字，返回排序后的结果。"

### Phase 1 复述
> 我的理解：实现一个排序函数，接收数字集合，返回升序排列的新集合。

### Phase 2 提取
| 维度 | 内容 |
|------|------|
| Goal | 实现排序函数 |
| Input | "一些数字" → 类型？容器？数量范围？ |
| Output | 排序后的结果 → 升序还是降序？新数组还是原地？ |
| Constraint | 无显式约束 |
| NFR | 无 |
| Context | 编程作业 → 可能有隐藏考察点 |

### Phase 3 澄清
| # | 分类 | 模糊点 | 处理 |
|---|------|--------|------|
| 1 | 🟡 | 数字类型 | 假设为整数（浮点排序逻辑相同） |
| 2 | 🟡 | 排序顺序 | 假设升序（最常见的默认） |
| 3 | 🟡 | 容器类型 | 假设为数组/列表 |
| 4 | 🟡 | 是否原地排序 | 假设返回新数组（更安全） |
| 5 | 🟢 | 算法选择 | 不限，但应讨论复杂度 |
| 6 | 🟢 | 边界情况 | 实现时自行处理（空/单元素/重复/负数） |
| 7 | 🔴 | **是否允许使用内置 sort？** | 这是编程作业的关键！ |

### Phase 4 分解
```
实现排序函数
├── ⭐ P0: 确认算法要求（手写 vs 内置 sort）
├── ⭐ P0: 实现排序逻辑
│   ├── 如果允许内置：调用 arr.sort()
│   └── 如果手写：
│       ├── 选择算法（快排/归并/堆排序）
│       ├── 实现 partition / merge / heapify
│       └── 处理边界情况
├── ⭐ P1: 编写测试用例
│   ├── 正常情况
│   ├── 空数组 / 单元素
│   ├── 已排序 / 逆序
│   ├── 含重复元素
│   └── 含负数
└── ⭐ P2: 优化（时间/空间复杂度分析）
```

### Phase 5 计划
1. **立即**：向用户确认 🔴 问题（是否允许内置 sort）
2. **待确认后**：编码实现 + 测试
3. **风险**：如果要求手写，算法选择会影响后续所有工作
