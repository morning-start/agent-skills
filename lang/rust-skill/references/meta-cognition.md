# Rust Skills 元认知框架

## 核心概念

不要直接回答，先通过认知层级追溯问题根源。

```
用户问题
    │
    ▼
┌─────────────────────────────────────────┐
│         Layer 1: 语言机制 (HOW)           │
│  Rust 语言特性和编译器规则                 │
│  例如: E0382 是所有权设计问题的症状        │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│         Layer 2: 设计选择 (WHAT)          │
│  设计模式和架构决策                       │
│  例如: 使用 Arc<T> 共享不可变数据         │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│         Layer 3: 领域约束 (WHY)           │
│  领域规则决定设计选择                      │
│  例如: 金融系统需要不可变、可审计的数据    │
└─────────────────────────────────────────┘
    │
    ▼
领域正确的架构解决方案
```

## 三层模型详解

### Layer 3: 领域约束 (Domain Constraints)

**核心问题**: 为什么这个数据需要这样设计？

领域规则决定了设计选择，不同领域有不同的约束：

- **金融系统**: 需要不可变、可审计的数据
- **ML 系统**: 需要内存效率和 GPU 加速
- **Web 服务**: 需要无状态、低延迟

### Layer 2: 设计选择 (Design Choices)

**核心问题**: 什么设计模式适合这个场景？

设计模式和架构决策：

- 使用 `Arc<T>` 共享不可变数据
- 使用 `Box<T>` 堆分配
- 选择同步还是异步

### Layer 1: 语言机制 (Language Mechanics)

**核心问题**: 如何用 Rust 语法实现？

Rust 语言特性和编译器规则：

- 所有权和借用规则
- 生命周期标注
- Trait 和泛型

## 路由规则

| 用户信号 | 入口层 | 追溯方向 | 主要技能 |
|---------|--------|---------|---------|
| E0xxx 错误 | Layer 1 | 向上追溯 ↑ | m01-m07 |
| "如何设计..." | Layer 2 | 双向 | m09-m15 |
| "[领域] 开发" | Layer 3 | 向下追溯 ↓ | domain-* |
| 性能问题 | Layer 1→2 | 先上后下 | m10-performance |

## 示例: E0382 错误

**传统 AI 回答**:
```
用户: 我的交易系统报 E0382
AI: 使用 .clone()
```

**元认知框架回答**:
```
用户: 我的交易系统报 E0382

AI (使用 Rust Skills):
├── Layer 1: E0382 = 所有权错误 → 为什么需要这个数据?
│       ↑
├── Layer 3: 交易记录是不可变的审计数据 → 应该共享而非复制
│       ↓
├── Layer 2: 使用 Arc<TradeRecord> 作为共享不可变值
│       ↓
└── 建议: 重设计为 Arc<T>, 而非 clone()
```

## 应用指南

当用户提出 Rust 相关问题时：

1. **识别入口层**: 根据错误代码或问题描述确定入口
2. **追溯方向**: 沿着认知层级向上/向下追溯
3. **多技能协作**:必要时组合多个技能提供完整方案
4. **提供领域感知**: 考虑用户的具体应用场景
