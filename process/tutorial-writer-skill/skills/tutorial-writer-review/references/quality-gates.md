# 质量门禁完整清单

> 本文档定义了教程撰写完成后必须通过的全部质量检查项，是 DELIVER 阶段的强制标准。

---

## Q1: 规划先行（PLAN）

### 检查方法

```
是否存在以下任一产物？
  ├─ 独立的规划文档（.md 文件）
  ├─ AGENTS.md 中批准的计划文件路径
  └─ 用户明确确认的"开始执行"指令
```

### 不通过的典型表现

- 直接从零开始写，没有任何前置规划
- 规划存在但未获用户确认就进入 WRITE
- 规划与最终产出内容完全无关

### 修正措施

回退到 PLAN 阶段，输出结构草图并获得确认后再继续。

---

## Q2: 搜索先行（SEARCH）

### 检查方法

```
是否满足以下全部条件？
  ├─ reference/ 对应目录下新增了 ≥ 1 个资料文件
  ├─ 资料文件内容与本章主题相关
  └─ 00-资源索引与链接.md 已更新（如有新增）
```

### 不通过的典型表现

- reference/ 目录为空或没有新增任何文件
- 资料文件内容与本章主题无关
- 仅查阅了旧资料但没有搜索新资料（对于新章节）

### 修正措施

回到 RESEARCH 阶段，执行完整的搜索→筛选→归档→索引流程。

---

## Q3: 长度达标（LENGTH）

### 检查方法

```
实际行数 vs 目标行数：

  │ 实际 < 目标×0.8  → 偏短，可能内容不够充实
  │ 目标×0.8 ≤ 实际 ≤ 目标×1.2  → 合格 ✅
  └ 实际 > 目标×1.2  → 偏长，需精简或拆分
```

### 各阶段容忍区间

| 阶段 | 目标行数 | 下限 (×0.8) | 上限 (×1.2) |
|------|---------|-------------|-------------|
| 认知篇 | 200-300 | 160-240 | 240-360 |
| 基础篇 | 350-500 | 280-400 | 420-600 |
| 进阶篇 | 250-350 | 200-280 | 300-420 |
| 生产篇 | 250-400 | 200-320 | 300-480 |
| 前沿篇 | 180-280 | 144-224 | 216-336 |

### 修正措施

- **偏短**：检查是否有遗漏的小节或要点，补充内容
- **偏长**：识别冗余段落精简，或将超长小节拆分为独立子文件

---

## Q4: 资料支撑（SOURCE）

### 检查方法

```
文中出现的以下元素是否都有资料支持？
  ├─ 具体数字（百分比、统计数据、版本号等）
  ├─ 对比结论（"X 比 Y 好 Z%"）
  ├─ 权威引用（"据 Gartner 报告..."）
  └─ 技术论断（非通用知识的声明）
```

### 不通过的典型表现

- 出现未标注来源的具体数字
- "研究表明"后面没有具体引用
- 技术参数凭记忆编写（可能与最新版本不符）

### 修正措施

回到 RESEARCH 阶段补充相关资料，或在文中将不确定的表述改为模糊表述并标注"待验证"。

---

## Q5: 结构对齐（ALIGN）

### 检查方法

```
逐项对比：
  ├─ 章节标题是否与大纲一致？
  ├─ 小节数量和标题是否与规划一致？
  ├─ 各小节的核心内容是否覆盖了规划的要点？
  └─ 是否有超出规划的游离内容？
```

### 允许的偏差

- 小节标题措辞微调（不影响含义）
- 因篇幅原因合并/拆分次要要点（需保持覆盖完整性）
- 增加衔接性过渡段落

### 不允许的偏差

- 缺失大纲中规定的小节
- 增加大纲中没有的全新主题
- 章节定位与大纲描述不符

---

## Q6: 语言规范（LANG）

### 检查项清单

```
□ 全文使用目标语言（中文）
□ 无代码注释风格的文字（# 这是说明）
□ 无占位符（TODO / 待补充 / xxx）
□ 无面向开发者的元说明
□ 术语首次出现时有英文原文标注
□ 全文术语使用一致（不混用"检索增强生成"/"RAG"/"检索生成"）
```

---

## Q7: 引用标注（CITE）

### 检查方法

全文扫描以下模式并验证其有效性：

```
（来源: ...）  →  对应的 reference/ 文件是否存在？
[来源X]        →  来源编号是否在上下文中定义？
```

### 常见引用错误

| 错误类型 | 示例 | 修正 |
|---------|------|------|
| 引用了不存在的文件 | （来源: 不存在的文件.md） | 删除或更正为正确路径 |
| 引用格式不规范 | "来自某网站" | 改为标准格式 |
| 遗漏引用 | 直接写出数字无来源 | 补充引用标注 |

---

## Q8: 格式统一（FMT）

### 检查项清单

```
□ 所有 ``` 代码块都有语言标注
□ 表格头尾对齐，分隔符正确
□ 标题层级不超过 4 级
□ 列表符号统一（同级同风格）
□ 空行使用合理（段落间有空行，列表内无多余空行）
```

## Q10: 跨章技术栈一致性（TECH）★ v2.1 新增

> **来源问题**：RAG 教程优化中发现 Ch3 主张"全栈原生实现"，Ch5 却使用 LangGraph，读者会产生"到底用不用框架"的困惑。

### 检查方法

```
1. 读取 decision-record.json 中的 tech_framework_preference 等技术选型决策
2. Grep 搜索本章中所有框架/库引用
3. 逐项对比是否与决策一致：
   ├─ 编排框架是否与全书选型一致？
   ├─ HTTP 库是否与全书统一（如 httpx vs requests）？
   ├─ 消息类型定义是否与前序章节兼容？
   └─ 如果引入了新框架，是否说明了与全书技术栈的关系？
```

### 不通过的典型表现

- 前序章节明确排除了某框架，本章却大量使用
- 同一功能在不同章节使用不同库（如 Ch3 用 requests，Ch5 用 httpx）
- 引入新框架但未说明与全书技术栈的关系

### 修正措施

统一为全书技术栈选型，或在章节开头明确说明"本章引入 XX 作为补充"。

## Q11: 跨章数据结构兼容（DATA）★ v2.1 新增

> **来源问题**：Ch4 输出 `{"text": str}` 格式，Ch5 使用 `chunk.page_content`（LangChain 格式），导致代码无法衔接运行。

### 检查方法

```
检查数据在章节间的流转：
  ├─ 上一章的输出数据格式是什么？（如 dict、dataclass、NamedTuple）
  ├─ 本章的输入假设是什么格式？
  ├─ 两者是否兼容？
  └─ 如果本章定义了新的数据结构，是否在章节开头明确声明？
```

### 不通过的典型表现

- 上一章输出 dict 格式，本章按对象属性访问（如 `chunk.page_content`）
- 数据流转的章节间缺少格式声明
- 数据格式变化后未同步更新后续章节

### 修正措施

统一数据格式，或在涉及数据流转的章节开头增加格式声明 blockquote。

---

## Q12: 大纲要点全覆盖（COVERAGE）★ v2.1 新增

> **来源问题**：多个章节遗漏了大纲中规划的要点，如 Ch5 缺 MMR/子问题分解，Ch12 缺音频/视频 RAG，Ch13 缺医疗案例。

### 检查方法

```
逐项对比大纲与正文：
  ├─ 大纲中每个要点在正文中是否有对应段落？
  ├─ 是否有正文内容超出了大纲范围？（允许适度扩展）
  ├─ 遗漏的要点是否有合理理由？（如标注为"进阶选读"）
  └─ 新增的正文内容是否在大纲中有对应位置？（如新增小节）
```

### 允许的偏差

- 小节标题措辞微调
- 要点合并/拆分（保持覆盖完整性）
- 增加衔接性过渡内容
- 标注为"选读/前沿扩展"的可选内容

### 不允许的偏差

- 大纲明确规划的要点完全缺失
- 章节定位与大纲描述严重不符

### 修正措施

补充遗漏要点，或将遗漏原因标注为"进阶选读，详见附录"。

---

## Q13: 技术深度达标（DEPTH）★ v2.1 新增

> **来源问题**：用户反馈"技术介绍要有深度，让读者知其然，知其所以然"。优化前多章技术介绍仅停留在"怎么用"层面。

### 检查方法

```
对每个核心技术点检查 S5 六维模型覆盖度：
  ├─ ① 概念定义：是否解释了"是什么"？
  ├─ ② 核心原理：是否说明了底层机制/公式？
  ├─ ③ 解决问题：是否解释了"为什么需要"？
  ├─ ④ 优缺点：是否给出了对比分析？
  ├─ ⑤ 发展历程：是否提到了起源和现状？
  └─ ⑥ 注意事项：是否列出了常见陷阱？

通过标准：核心概念至少覆盖 4/6 维度
```

### 不通过的典型表现

- 技术介绍仅有一行定义 + 代码示例，缺少"为什么需要"的解释
- 多个技术点的介绍深度严重不均（有的深入，有的仅一句话）
- 缺少优缺点对比，读者无法判断技术选型

### 修正措施

按照 S5 六维模型补充缺失维度，确保核心概念有足够深度。

## Q14: 网页构建通过（WEB）★ v2.2 新增

> 仅在启用了网页发布流程时检查。纯 Markdown 交付可跳过此项（标记为 N/A）。

### 检查方法

```
执行 mkdocs build 并检查输出：
  ├─ 是否有 error？ → 阻塞，修复后重建
  ├─ 是否有 warning？ → 建议修复
  ├─ 所有图片路径在 site/ 目录下可访问？
  └─ 自定义 JS/CSS 文件路径有效？
```

### 不通过的典型表现

- `mkdocs build` 报 `Error` 退出（通常为路径错误、语法错误）
- 构建成功但某些页面在浏览器中显示 404 或资源加载失败
- Mermaid 图表在 `mkdocs serve` 预览中不渲染

### 修正措施

```bash
# 本地构建验证
uv run mkdocs build --clean

# 常见修复
# ─ 图片路径: 确保引用路径以 ./ 或 / 开头，指向 docs/ 内
# ─ Mermaid 语法: 用 https://mermaid.live 在线验证
# ─ 导航配置: 检查 mkdocs.yml 中 nav 指向的文件是否存在
```
