# AGENTS.md 类型分类体系

> 本文档详细定义了 5 种 AGENTS.md 类型，每种类型都有明确的适用场景、结构特征和判定标准。

---

## 类型总览

```
┌─────────────────────────────────────────────────────────────┐
│                  AGENTS.md 5 种类型                          │
├─────────────┬──────────┬──────────┬─────────────────────────┤
│ 类型         │ 复杂度     │ 行数范围   │ 典型场景                │
├─────────────┼──────────┼──────────┼─────────────────────────┤
│ A: 最小配置型  │ 0-20     │ 30-80    │ 个人脚本/实验性项目         │
│ B: 标准手册型  │ 21-40    │ 80-200   │ 小型库/独立应用            │
│ C: 框架技能型  │ 41-60    │ 200-400  │ 中型框架/工具链/技能库       │
│ D: 领域认知型  │ 61-80    │ 400-600  │ 复杂领域项目/多模块系统      │
│ E: 系统集成型  │ 81-100   │ 600-1000 │ 企业级/大型开源/高合规       │
└─────────────┴──────────┴──────────┴─────────────────────────┘
```

---

## Type A: 最小配置型

### 定义
最精简的 AGENTS.md，仅包含核心要素，适合个人或极简项目。

### 适用场景
- ✅ 个人工具、脚本、自动化任务
- ✅ 实验性项目、原型、MVP
- ✅ 学习练习、教学示例
- ✅ 一次性使用的代码库

### 不适用场景
- ❌ 团队协作项目（> 1人长期维护）
- ❌ 需要对外发布或开源的项目
- ❌ 有复杂技术栈或多模块依赖的项目

### 结构特征

```yaml
必需组件:
  - 前言区 (YAML): 5-8 行
  - 身份与角色: 3-5 行
  - 核心规则: 5-10 行（3-5条规则）
  - 工作流: 5-8 行（简化版 3 步骤）
  - 注意事项: 3-5 行

可选组件:
  - 快速命令参考: 5-10 行（如有常用命令）

总计: 26-46 行（必需）+ 0-10 行（可选）
```

### 标准模板

```yaml
---
name: {project}-agents
version: v1.0.0

description: {一句话描述}
tags: [{tag1}, {tag2}]

## 触发条件

## 核心工作流
```
理解需求 → 制定方案 → 执行实施 → 验证结果
```

## 核心规则
| # | 规则 | 说明 |
|---|------|------|
| R1 | {rule_1} | {detail} |
| R2 | {rule_2} | {detail} |

```bash
{build_command}

```

## 注意事项
⚠️ {most_important_thing}

### 真实案例

**案例 1: 个人备份脚本**

```yaml
---
name: backup-script-agents
version: v1.0.0
author: lucifer
description: 自动备份脚本的Agent配置 — 简单的文件同步和压缩任务
tags: [backup, bash, automation]
---

# 备份脚本 Agent 配置

## 身份与角色
你是 Bash 脚本专家，专注于文件备份和归档任务。

## 触发条件
当用户需要以下操作时激活：
- 创建或修改备份脚本
- 调试备份失败问题
- 优化备份性能

## 核心工作流
```
① 分析源目录和目标位置
② 选择备份策略（增量/全量）
③ 编写或修改脚本
④ 测试运行（使用 --dry-run）
⑤ 交付可执行脚本
```

## 核心规则
| # | 规则 | 说明 |
|---|------|------|
| R1 | 使用 `set -euo pipefail` | 严格模式，错误即停止 |
| R2 | 所有路径必须用引号包裹 | 处理含空格的文件名 |
| R3 | 备份前必须检查磁盘空间 | 至少保留 20% 可用空间 |
| R4 | 日志输出到 stderr | stdout 保留给数据流 |

## 常用命令
```bash
# 测试脚本
./backup.sh --dry-run --verbose

# 执行备份
./backup.sh --source ~/Documents --target /mnt/backup

# 查看日志
tail -f /var/log/backup.log
```

## 注意事项
⚠️ **安全第一**: 永远先 `--dry-run` 测试，再实际执行
⚠️ **权限意识**: 避免使用 sudo，除非绝对必要
⚠️ **数据验证**: 备份后随机抽样校验文件完整性
```

**规模**: 52 行  
**类型评分**: 12/100 (Type A)

## Type B: 标准手册型

### 定义
结构完整的 AGENTS.md，覆盖主要使用场景，适合小型正式项目。

### 适用场景
- ✅ 小型库、包、SDK
- ✅ 独立应用（Web/CLI/GUI）
- ✅ 2-5 人团队协作
- ✅ 初创公司的内部工具

### 不适用场景
- ❌ 大型框架或平台
- ❌ 多团队协作的企业系统
- ❌ 有复杂领域知识要求的项目

### 结构特征

```yaml
必需组件:
  - 前言区 (YAML): 5-8 行
  - 身份与角色: 5-8 行
  - 触发条件: 3-5 行
  - 意图路由表: 10-15 行（3-5个任务）
  - 通用工作流: 10-15 行（5步标准流程）
  - 编码规范: 10-15 行（5-8条规则）
  - 常用命令: 8-12 行（分类组织）
  - 注意事项: 5-8 行

可选组件:
  - 项目架构概览: 10-15 行
  - 第三方依赖说明: 5-8 行

总计: 61-94 行（必需）+ 15-23 行（可选）
```

### 与 Type A 的关键差异

| 维度 | Type A | Type B |
|------|--------|--------|
| **意图路由** | 无或极简（< 5行） | 完整的路由表（10-15行） |
| **编码规范** | 3-5条简单规则 | 5-8条规则 + 格式化要求 |
| **工作流** | 3步简化版 | 5步标准版 + 异常处理 |
| **错误处理** | 仅注意事项 | 独立章节（常见错误表） |
| **架构说明** | 无 | 可选章节 |

# React UI 组件库 Agent 配置

## 身份与角色
你是 React UI 组件库的开发专家，专注于：
- 设计和实现高质量的 React 组件
- 编写 Storybook 文档和演示
- 确保组件的可访问性和响应式设计
- 维护组件的 TypeScript 类型定义

### 工作原则
1. **Composition over Inheritance**: 组合优于继承
2. **Controlled Components**: 受控组件模式
3. **Accessibility First**: 无障碍优先
4. **Consistent API**: 组件 API 风格统一

## 触发条件
当用户需要以下操作时激活：
- 创建新组件或修改现有组件
- 编写或更新 Storybook stories
- 调试组件样式或行为问题
- 发布新版本的组件库

**不激活的场景**: 纯业务逻辑开发（非 UI 相关）

## 意图路由

| 用户意图 | 触发示例 | 执行动作 | 优先级 |
|---------|---------|---------|--------|
| **🧩 创建组件** | "新建Button"/"写个Modal" | → 执行 [create-component](#创建组件流程) | P0 |
| **📝 编写文档** | "写Story"/"更新docs" | → 进入 Storybook 模式 | P0 |
| **🧪 测试组件** | "测试组件"/"coverage" | → 执行 [test-guide](#测试策略) | P0 |

| **📦 发布版本** | "发布"/"npm publish" | → 执行 [publish-flow](#发布流程) | P2 |

## 通用工作流

所有任务遵循 **DESIGN → IMPLEMENT → DOCUMENT → TEST → RELEASE** 流程：

```
① DESIGN（设计）
   - 分析需求和 UX 要求
   - 确定 Props 接口（TypeScript）
   - 设计状态管理和回调

② IMPLEMENT（实现）
   - 创建组件文件（src/components/Xxx/index.tsx）
   - 实现 render 逻辑和样式
   - 导出类型定义

③ DOCUMENT（文档）
   - 编写 Storybook story (.stories.tsx)
   - 添加 Props 文档注释
   - 创建使用示例

④ TEST（测试）
   - 编写单元测试（@testing-library/react）
   - 测试可访问性（axe-core）
   - 验证视觉回归（如果配置了）

⑤ RELEASE（发布）
   - 更新 CHANGELOG.md
   - 执行版本 bump
   - 发布到 npm
```

## 编码规范

### 必须遵守的规则

| # | 规则 | ✅ 正确 | ❌ 错误 | 检测方式 |
|---|------|--------|--------|---------|
| R1 | 函数式组件 + Hooks | `<Component />` | `class Component` | ESLint 规则 |
| R2 | TypeScript 严格模式 | `interface Props` | `any` 类型 | tsc --strict |
| R3 | Props 解构 | `({ label, onClick })` | `props.label` | Code Review |
| R4 | CSS Modules 或 Styled | `styles.button` | 内联 style | Stylelint |
| R5 | 导出类型 | `export type Props` | 仅导出组件 | tsc 检查 |

### 命名约定
| 类型 | 格式 | 示例 |
|------|------|------|
| 组件文件 | PascalCase.tsx | `Button.tsx` |
| 样式文件 | *.module.css | `Button.module.css` |
| Story 文件 | *.stories.tsx | `Button.stories.tsx` |
| 测试文件 | *.test.tsx | `Button.test.tsx` |
| 类型导出 | types.ts | `Button/types.ts` |

## 常见错误及解决方案

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Storybook 加载失败 | 缺少 `.stories.tsx` 文件 | 检查文件命名和导出 |
| 样式不生效 | CSS Modules 冲突 | 使用唯一的 class 名 |
| TypeScript 报错 | Props 类型缺失 | 补充完整的接口定义 |
| 测试 flaky | 异步状态未处理 | 使用 `act()` 包裹 |

## Type C: 框架技能型

### 定义
功能丰富的 AGENTS.md，具备子技能调度能力，适合中型框架或技能库。

### 适用场景
- ✅ 中型框架、工具链、CLI 工具
- ✅ 技能库（如 book-skills、moonbit-skills）
- ✅ 开源项目（中等规模的社区项目）
- ✅ 5-15 人的团队

### 不适用场景
- ❌ 简单的单体应用
- ❌ 企业级多系统集成
- ❌ 颀域知识密集的专业系统

### 结构特征

```yaml
必需组件:
  - 前言区 (YAML): 5-8 行
  - 身份与角色: 5-8 行
  - 触发条件: 5-8 行
  - 意图路由表: 20-30 行（5-8个任务，含子任务）
  - 通用工作流: 15-20 行（含分支逻辑）
  - 子技能/模块索引: 10-15 行
  - 编码规范: 15-20 行（含具体规则和示例）
  - 测试策略: 8-12 行
  - 常用命令: 10-15 行（按场景分类）
  - 错误处理指南: 10-15 行
  - 注意事项: 8-12 行

可选组件:
  - 架构决策记录: 10-15 行
  - 性能优化建议: 8-12 行
  - 与其他工具集成: 5-10 行

总计: 111-171 行（必需）+ 33-47 行（可选）
```

### 与 Type B 的关键差异

| 维度 | Type B | Type C |
|------|--------|--------|
| **子技能** | 无 | 有 skills/ 子目录 |
| **意图路由** | 平铺式 | 分层（P0/P1/P2）+ 条件判断 |
| **工作流** | 线性 5 步 | 含分支和异常处理 |
| **测试** | 简单提及 | 独立章节（策略+工具+覆盖率） |
| **错误处理** | 常见表 | 诊断流程 + 错误码映射 |

  - 团队协作规范: 10-15 行

可选组件:
  - 灾难恢复计划: 10-15 行
  - 性能SLA定义: 8-12 行
  - 外部依赖管理: 10-15 行
  - 知识产权声明: 5-8 行

总计: 378-497 行（必需）+ 48-55 行（可选）
```

## 安全与合规

### 数据保护

- 审批流程: 生产环境变更需 2 人审批

### 安全扫描要求
- SAST: 每次提交自动扫描
- DAST: 每周动态扫描
- 依赖审计: 每日检查 CVE
```

## 类型演进路线

项目会随时间演化，AGENTS.md 类型也应随之升级：

```
Type A ──→ Type B ──→ Type C ──→ Type D/E
(个人)     (小团队)    (成长中)     (成熟期)

典型触发事件:
A → B: 第二人加入 / 决定开源
B → C: 功能模块化 / 引入子技能
C → D: 领域深化 / 复杂度激增
C/D → E: 企业级合规要求 / 大规模集成
```

**升级时机建议**:
- 每季度重新评估一次
- 重大架构变更后立即评估
- 团队规模翻倍时考虑升级
