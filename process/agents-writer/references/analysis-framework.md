# 五维分析框架

> 用于阶段 1（1-analyze）的完整方法论。每个维度的评估指标、数据来源、评分标准。

---

## 维度 1: 产品定位

### 评估指标

| 指标 | 数据来源 | 评估方法 |
|------|---------|---------|
| 一句话定位 | README 前 3 段 / 项目网站 | 提取项目的核心价值主张 |
| 解决场景 | README 的 "Features" 或 "Use Cases" 章节 | 列表化目标场景 |
| 目标用户 | README 的 "Who is this for" 或隐式推断 | 根据文档语气和粒度判断 |
| 竞品/替代方案 | README 的对比章节或用户描述 | 了解为什么用户选择这个项目 |
| 项目阶段 | commit 频率/版本号/发布日志 | 判断是早期还是成熟期 |

### 评分标准

| 分数 | 描述 | 典型特征 |
|------|------|---------|
| 1 | 不清楚项目做什么 | README 没有明确的价值主张 |
| 2 | 大概知道方向 | 有描述但模糊 |
| 3 | 定位清晰 | 有明确的一句话定位 |
| 4 | 定位+差异化 | 清楚自己的竞争优势 |
| 5 | 完整产品认知 | 定位 + 差异化 + 目标市场 + 阶段 |

## 维度 2: 目标用户

### 评估指标

| 指标 | 数据来源 | 评估方法 |
|------|---------|---------|
| 技术水平 | README 风格、API 文档深度 | 新手友好还是专业术语密集 |
| 团队规模 | 贡献者数、组织归属 | git shortlog、package.json 的 author |
| 使用频率 | 项目类型推断 | 库/工具（高频）vs 脚手架（低频） |
| Agent 角色 | 项目复杂度推断 | 单 Agent 还是多 Agent 协作 |

### 评分标准

| 分数 | 描述 | 典型特征 |
|------|------|---------|
| 1 | 用户画像完全不清楚 | 无法推断使用场景 |
| 2 | 大概知道谁在用 | 有目标用户但特征模糊 |
| 3 | 用户画像清晰 | 明确知道使用者的身份 |
| 4 | 多角色识别 | 能区分开发者/运维/最终用户 |
| 5 | 完整的 user journey | 知道用户的使用流程和接触点 |

---

## 维度 3: 功能边界

### 评估指标

| 指标 | 数据来源 | 评估方法 |
|------|---------|---------|
| 核心功能 | 代码结构 / features 目录 | 提取主要功能模块 |
| 文件权限 | 目录结构、.gitignore、权限配置 | 识别哪些文件 Agent 可以/不可以操作 |
| 命令白名单 | Makefile / package.json scripts | 提取可执行命令 |
| 命令黑名单 | 安全敏感操作 | 识别可能危险的命令 |

### 评分标准

| 分数 | 描述 | 典型特征 |
|------|------|---------|
| 1 | 未识别任何边界 | 不知道项目哪些部分可以操作 |
| 2 | 识别了核心功能 | 知道主要功能 A 和 B |
| 3 | 功能 + 文件权限 | 知道哪些文件能碰、哪些不能 |
| 4 | 完整边界映射 | 功能 + 文件 + 命令 + 配置覆盖 |
| 5 | 精细化权限控制 | 场景化的边界（不同角色不同权限） |

---

## 维度 4: 安全检查

### 评估指标

| 指标 | 数据来源 | 评估方法 |
|------|---------|---------|
| 数据敏感性 | 隐私政策 / 数据库结构 | 是否有用户数据、PII |
| 凭证管理 | .env / .gitignore / 配置 | 是否有 API 密钥、密码 |
| 部署配置 | docker-compose / CI / CD | 是否有生产环境配置 |
| 合规要求 | LICENSE / 公司制度 | 是否有合规要求 |
| 自动化测试 | tests/ 目录、CI 流程 | 测试覆盖率是否完备 |

### 评分标准

| 分数 | 描述 | 典型特征 |
|------|------|---------|
| 1 | 未检查安全性 | 不知道项目安全需求 |
| 2 | 基础安全认知 | 知道有 API 密钥但未制定规则 |
| 3 | 安全检查清单 | 有安全红线的初步识别 |
| 4 | 场景化安全规则 | 不同操作有不同的安全门禁 |
| 5 | 完整安全体系 | 红线 + 门禁 + 合规 + 审计追溯 |

---

## 维度 5: 架构规划

### 评估指标

| 指标 | 数据来源 | 评估方法 |
|------|---------|---------|
| 架构模式 | 目录结构 / 代码阅读 | 模式识别（MVC/分层/微服务） |
| 技术栈 | package.json / Cargo.toml | 列出所有技术依赖 |
| 模块组织 | 目录树的组织方式 | 理解包/模块划分逻辑 |
| 构建发布 | CI 配置 / Makefile | 完整的构建流程 |
| 测试策略 | 测试目录分布 | 单元/集成/E2E 的覆盖情况 |

### 评分标准

| 分数 | 描述 | 典型特征 |
|------|------|---------|
| 1 | 架构完全不清楚 | 不知道代码怎么组织的 |
| 2 | 知道技术栈 | 知道用什么语言和框架 |
| 3 | 理解目录结构 | 能描述主要目录的职责 |
| 4 | 掌握依赖关系 | 理解模块间的调用关系和依赖方向 |
| 5 | 全貌掌握 | 架构模式 + 技术栈 + 模块依赖 + 构建测试全覆盖 |
