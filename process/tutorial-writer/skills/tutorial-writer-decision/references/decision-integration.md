# 决策系统集成指南

> 本文档定义了决策记录系统在 PRWRD 工作流中的集成方式，包括架构定位、各阶段检查点、质量门禁扩展和快速启动指南。

## 系统架构概览

### 核心目标

将书籍创作过程中的**非大纲类关键决策**系统化管理，确保：
- ✅ 所有重要决策有据可查、有迹可循
- ✅ Agent 能主动发起讨论并引导用户做出选择
- ✅ 决策可随时查看、修改和追溯历史
- ✅ 决策变更影响范围可控，避免遗漏

### 系统定位

```
┌─────────────────────────────────────────────────────┐
│                  PRWRD 工作流                        │
│                                                     │
│  PLAN → RESEARCH → WRITE → REVIEW → DELIVER        │
│         ↑                                           │
│         │                                           │
│  ┌──────┴──────┐                                    │
│  │  DECISION   │ ← 新增阶段（在 PLAN 之前或并行）    │
│  │  RECORD     │                                    │
│  └─────────────┘                                    │
│                                                     │
│  决策记录贯穿整个生命周期：                            │
│  - PLAN 阶段：读取决策指导规划                        │
│  - WRITE 阶段：遵循决策执行撰写                       │
│  - REVIEW 阶段：验证内容符合决策                      │
└─────────────────────────────────────────────────────┘
```

### 与现有工作流的关系

| 工作流阶段 | 决策记录的作用 | 调用时机 |
|-----------|--------------|---------|
| **DECISION（新增）** | 收集/确认/更新决策 | 项目启动时、需求变化时 |
| **PLAN** | 读取已确认决策，指导章节规划 | 每章开始前 |
| **RESEARCH** | 根据技术选型确定搜索方向 | 搜索关键词设计 |
| **WRITE** | 遵循风格/语言/深度等决策 | 撰写过程中 |
| **REVIEW** | 检验输出是否符合决策约定 | 质量检查时 |
| **DELIVER** | 记录本次交付依赖的决策版本 | 交付总结中 |

## 集成到 PRWRD 工作流

### 修订后的完整工作流

```
┌─────────────────────────────────────────────────────────────┐
│                   PRWRD+D 工作流（增强版）                    │
│                                                             │
│  DECISION → PLAN → RESEARCH → WRITE → REVIEW → DELIVER     │
│                                                             │
│  新增的 DECISION 阶段：                                       │
│  - 位于 PLAN 之前（项目启动时）                               │
│  - 可与 PLAN 并行（章节级决策）                              │
│  - 在任何阶段都可回溯（检测到缺失/冲突时）                    │
└─────────────────────────────────────────────────────────────┘
```

### 各阶段的决策检查点

| 阶段 | 入口检查 | 出口动作 | 异常处理 |
|------|---------|---------|---------|
| **DECISION** | 加载/初始化决策记录 | 输出决策状态报告 | 缺少必要决策 → 发起讨论 |
| **PLAN** | 读取相关决策（影响本章的） | 规划符合决策约束 | 决策冲突 → 暂停并解决 |
| **RESEARCH** | 根据技术选型调整搜索策略 | 资料匹配技术栈 | 技术栈模糊 → 先确认再搜索 |
| **WRITE** | 实时遵循风格/语言决策 | 内容符合所有适用决策 | 发现新决策点 → 补充并讨论 |
| **REVIEW** | 验证输出 vs 决策约定 | 质量门禁+决策合规检查 | 不符合 → 修正或标记例外 |
| **DELIVER** | 记录使用的决策版本 | 交付物附带决策快照 | 版本不一致 → 提示用户 |

### 质量门禁扩展

在原有门禁基础上，新增**决策合规检查**：

| # | 检查项 | 缩写 | 级别 | 通过标准 |
|---|--------|------|------|---------|
| Q10 | **决策完备性** | DEC-COMPLETE | P0-阻塞 | 本章涉及的所有 critical/high 决策已确认 |
| Q11 | **决策一致性** | DEC-CONSIST | P1-警告 | 内容与已确认决策无矛盾 |
| Q12 | **决策引用** | DEC-CITE | P2-建议 | 关键决策点在章节中有体现或说明 |

**更新后的通过条件**：
- Q1-Q2 必须全部通过（原有阻塞项）
- **Q10 必须通过（新增阻塞项）**
- Q3-Q9, Q11-Q12 至少 9/10 项通过

## 快速启动指南

### 首次使用（项目初始化）

```bash
# 1. 复制模板到项目目录
cp .trae/skills/tutorial-writer/assets/decision-record-template.json ./decision-record.json

# 2. Agent 自动加载并检查状态
# （Agent 在首次 PLAN 前会自动执行）

# 3. Agent 发起首批 critical 决策讨论
# 通常包括：
#   - basic_lang_primary (主要语言)
#   - tech_primary_language (编程语言)
#   - content_target_audience (目标用户)
#   - project_quality_standard (质量标准)
```

### 常用命令速查

```bash
# 查看决策状态总览
cat decision-record.json | jq '.metadata'

# 查看特定决策
cat decision-record.json | jq '.decisions.tech_primary_language'

# 查看已确认决策数量
cat decision-record.json | jq '.metadata.confirmed_decisions'

# 查看最近一次讨论
cat decision-record.json | jq '.discussion_sessions[-1]'

# 查看版本历史
cat decision-record.json | jq '.version_history'
```

### Agent 调用示例

**场景 1：项目启动时自动发起讨论**

```
User: "开始写 RAG 教程"

Agent:
  1. 检测到 decision-record.json 存在但全部 pending
  2. 加载决策记录到 Memory
  3. 发起 DECISION 阶段:

     "🚀 在正式开始写作前，我需要先和您确认几个关键决策。
      这些决策将影响整本书的风格、技术栈和内容深度。

      今天我们先讨论最关键的 4 个决策（预计 5-10 分钟）：

      1. 📝 写作语言
      2. 💻 编程语言
      3. 👥 目标读者
      4. 🎯 内容深度

      准备好了吗？我们从第一个开始..."
```

**场景 2：写作过程中检测到缺失决策**

```
User: "开始写 Ch4 数据处理"

Agent:
  1. 检查 Ch4 相关的决策
  2. 发现 tech_vector_db (向量数据库) 还是 pending
  3. 暂停 PLAN，插入讨论:

     "⏸️ 在规划 Ch4 之前，我需要先确认一个技术选型：

      Ch4 将讲解数据处理与向量化，需要用到向量数据库。
      目前还未确定使用哪一个。

      选项：
      A) ChromaDB - 最简单，5 行代码上手（推荐初学者）
      B) FAISS - 性能优秀，生产环境常用
      C) 两者都讲（ChromaDB 入门 + FAISS 进阶）

      您倾向哪个？"
```

**场景 3：用户要求查看/修改决策**

```
User: "看看目前都确定了哪些决策"

Agent:
  1. 读取 decision-record.json
  2. 生成可视化报告
  3. 展示给用户

User: "把图文比例改成图为主"

Agent:
  1. 执行修改流程
  2. 影响评估：影响全书排版密度
  3. 确认后执行修改
  4. 通知用户完成
```

## 注意事项与最佳实践

### ⚠️ 重要提醒

1. **不要过度讨论**
   - 不是所有 20 个决策都需要在一开始就全部确认
   - 采用"按需讨论"策略：用到时再确认
   - critical/high 优先级的尽量早期确认

2. **尊重用户选择**
   - Agent 给出推荐但不强求
   - 用户选择了非推荐选项也应尊重
   - 记录用户选择的 reasoning 以便追溯

3. **保持灵活性**
   - 决策不是一成不变的，允许合理变更
   - 变更时要充分评估影响
   - 重要变更建议获得用户二次确认

4. **避免决策疲劳**
   - 每次讨论控制在 2-4 个决策
   - 复杂决策拆分为多次讨论
   - 提供跳过/推迟选项

### ✅ 最佳实践

| 实践 | 说明 | 示例 |
|------|------|------|
| **批量讨论关联决策** | 依赖关系密切的一起讨论 | 语言+框架+数据库一次性确认 |
| **提供上下文** | 讨论时说明为什么需要这个决策 | "因为这会影响 Ch3-Ch5 的代码示例" |
| **记录决策理由** | 不仅记录 what，还要记录 why | 用户选 Python 是因为读者群体 |
| **定期同步** | 长项目期间定期回顾决策 | 每完成一篇后快速 review 决策 |
| **版本快照** | 重大节点保存决策快照 | Ch1 完成时保存 v1.0 版本 |

### 🚫 常见陷阱

| 陷阱 | 表现 | 规避方式 |
|------|------|---------|
| **决策膨胀** | 不断添加新决策，超过 30 个 | 严格控制，非必要不添加 |
| **分析瘫痪** | 过度讨论导致迟迟不动笔 | 设置讨论时限（15 分钟/批） |
| **幽灵决策** | 记录了但从不查阅 | 每个阶段入口强制检查 |
| **孤岛决策** | 修改了一个没通知相关方 | 依赖关系+级联更新机制 |
