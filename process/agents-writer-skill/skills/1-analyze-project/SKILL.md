# ① 项目分析与类型判定

## 任务目标

分析目标项目的特征，自动判定最适合的 AGENTS.md **类型** 和**复杂度等级**，为后续的结构设计和内容编写提供决策依据。

## 触发条件

当用户说以下任一话术时激活：
- "帮我分析这个项目"
- "什么类型的AGENTS.md适合"
- "应该写多复杂"
- "评估一下项目需求"

## 决策树

```
开始分析
    │
    ▼
收集项目信息（技术栈/团队/规模/协作模式）
    │
    ▼
┌─────────────────────────────────────┐
│  7维评估模型                         │
│  ┌──────────┬──────────┬──────────┐ │
│  │ 技术栈   │ 团队规模 │ 协作模式 │ │
│  │ 复杂度   │ 维护频率 │ Agent数量│ │
│  │ 特殊需求 │          │          │ │
│  └──────────┴──────────┴──────────┘ │
└─────────────────────────────────────┘
    │
    ▼
计算复杂度评分 (0-100)
    │
    ├── 0-20  → Type A: 最小配置型
    ├── 21-40 → Type B: 标准手册型
    ├── 41-60 → Type C: 框架技能型
    ├── 61-80 → Type D: 领域认知型
    └── 81-100→ Type E: 系统集成型
```

## 操作步骤

### Step 1: 收集项目信息（必做）

通过以下方式收集项目上下文：

```bash
# 检查项目基本信息
ls -la | head -20                    # 目录结构
cat README.md | head -50             # 项目说明
cat package.json 2>/dev/null || cat Cargo.toml 2>/dev/null || cat pyproject.toml 2>/dev/null  # 包管理

# 检查现有配置文件
ls -la .claude* .cursor* AGENTS.md 2>/dev/null  # 已有Agent配置
cat .gitignore | grep -E "(agent|ai|claude)" 2>/dev/null  # AI相关忽略规则

# 统计代码规模
find . -name "*.ts" -o -name "*.py" -o -name "*.rs" | wc -l  # 文件数量
wc -l $(git ls-files) 2>/dev/null | tail -1  # 总行数
```

**必须收集的 5 个关键信息**：

| # | 信息项 | 获取方式 | 权重 |
|---|--------|---------|------|
| 1 | 技术栈（语言/框架） | package.json / Cargo.toml | 20% |
| 2 | 项目规模（文件数/行数） | git ls-files / find | 25% |
| 3 | 团队规模（贡献者数） | git shortlog -sn | 15% |
| 4 | 是否有现有Agent配置 | ls .claude* .cursor* | 20% |
| 5 | 特殊需求（FFI/多平台/DSL） | README / 文档 | 20% |

### Step 2: 执行 7 维评估（核心）

对每个维度打分（0-10分），加权求和：

#### 维度 1: 技术栈复杂度 (权重 20%)

| 分数 | 场景 |
|------|------|
| 1-3 | 单一语言，无特殊框架（纯 JS / Python 脚本） |
| 4-6 | 主流框架（React / Django / Rust 标准库） |
| 7-8 | 多语言/多框架（全栈 / Wasm+JS 双后端） |
| 9-10 | 特殊技术栈（DSL / 编译器 / FFI 密集） |

**示例**：
- MoonBit 项目（Wasm+JS+Native 三后端）→ **9分**
- 普通 React 前端 → **5分**
- 纯 Python 脚本 → **2分**

#### 维度 2: 项目规模 (权重 25%)

| 分数 | 文件数 | 代码行数 |
|------|--------|---------|
| 1-3 | < 50 文件 | < 5,000 行 |
| 4-6 | 50-200 文件 | 5,000-20,000 行 |
| 7-8 | 200-500 文件 | 20,000-50,000 行 |
| 9-10 | > 500 文件 | > 50,000 行 |

#### 维度 3: 团队协作模式 (权重 15%)

| 分数 | 场景 |
|------|------|
| 1-3 | 个人项目 / 单人维护 |
| 4-6 | 小团队（2-5人），偶尔协作 |
| 7-8 | 中型团队（6-15人），频繁协作 |
| 9-10 | 大型团队（>15人）或多团队协作 |

#### 维度 4: 维护频率 (权重 10%)

| 分数 | commit 频率 |
|------|-----------|
| 1-3 | < 10次/月（低频维护） |
| 4-6 | 10-50次/月（中等活跃） |
| 7-8 | 50-200次/月（高频迭代） |
| 9-10 | > 200次/月（持续集成） |

#### 维度 5: Agent 使用深度 (权重 20%)

| 分数 | 场景 |
|------|------|
| 1-3 | 偶尔使用 AI 辅助（代码补全/简单问答） |
| 4-6 | 常规使用（日常编码/重构/测试生成） |
| 7-8 | 深度集成（CI/CD Agent / 自动化工作流） |
| 9-10 | 核心依赖（AI 驱动的开发流程 / 多 Agent 协作） |

#### 维度 6: 特殊架构需求 (权重 5%)

| 分数 | 场景 |
|------|------|
| 1-3 | 标准单体架构 |
| 4-6 | Monorepo / 微服务 |
| 7-8 | 多平台编译（Wasm/Native/Web） |
| 9-10 | 自定义 DSL / 编译器扩展 |

#### 维度 7: 合规与安全要求 (权重 5%)

| 分数 | 场景 |
|------|------|
| 1-3 | 无特殊要求 |
| 4-6 | 基本代码规范（ESLint / Clippy） |
| 7-8 | 安全审计 / 合规检查（HIPAA / GDPR） |
| 9-10 | 高安全域（金融 / 医疗 / 国防） |

### Step 3: 计算总分并匹配类型

```
总分 = Σ(维度分数 × 权重)

示例计算：
MoonBit-skills 项目:
  技术栈(9×0.20) + 规模(6×0.25) + 团队(4×0.15) +
  维护(7×0.10) + Agent深度(8×0.20) + 架构(7×0.05) + 合规(3×0.05)
= 1.8 + 1.5 + 0.6 + 0.7 + 1.6 + 0.35 + 0.15
= **6.7 (四舍五入到整数 7)** → 实际应该是 67 分
```

**类型匹配表**：

| 总分范围 | 推荐类型 | 典型场景 | 推荐行数 |
|---------|---------|---------|---------|
| **0-20** | **A: 最小配置型** | 个人脚本/工具项目 | 30-80 行 |
| **21-40** | **B: 标准手册型** | 小型库/独立应用 | 80-200 行 |
| **41-60** | **C: 框架技能型** | 中型框架/工具链 | 200-400 行 |
| **61-80** | **D: 领域认知型** | 复杂领域项目/多模块系统 | 400-600 行 |
| **81-100** | **E: 系统集成型** | 企业级/多团队/高合规 | 600-1000 行 |

> 📖 详细类型定义见 [type-classification.md](../references/type-classification.md)

## 常见问题

**Q: 如果项目同时符合多个类型怎么办？**

A: 取**最高分类型**，但在设计结构时可参考其他类型的优点。例如：
- 评分 55（接近 Type D）但团队很小 → 用 Type C 结构 + Type D 的部分高级特性

**Q: 已有 AGENTS.md 怎么办？**

A: 先用 [⑤ 质量检查](../skills/5-quality-check/SKILL.md) 评估现有文件，再决定是优化还是重写。

**Q: 评分边界值如何处理？**

A: **向下取整**更保守（例如 40 分 → Type B 而非 Type C），除非项目有明显的高复杂度特征。

## 注意事项

⚠️ **客观性原则**: 评分应基于可验证的事实（文件数、commit 记录），而非主观感受。

⚠️ **动态调整**: 项目会演化，建议每季度重新评估一次类型，特别是活跃项目。

⚠️ **保守策略**: 不确定时选择更低类型，避免过度设计。AGENTS.md 可以逐步增强，但过度复杂的初始版本会增加维护成本。
