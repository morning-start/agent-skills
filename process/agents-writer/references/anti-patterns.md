## 一、内容反模式（8 种）

### AP-01: 教程化倾向

**表现**：花大量篇幅介绍项目背景、语言特性、框架概念，而不是告诉 Agent 如何工作。

**错误示例** ❌：
```markdown
此项目使用 Rust 语言，Rust 是一门系统级编程语言，
以内存安全著称。它拥有所有权系统、生命周期标注、
Trait 等高级特性。项目中使用 Actix-web 框架...
```

**正确示例** ✅：
```markdown
## 行为规则
- 所有 API 端点遵循 `src/api/` 下的模块化组织方式
- 错误处理使用 `Result<T, AppError>`，不要使用 unwrap()
- 添加新路由前，先阅读 `src/api/mod.rs` 了解注册模式
```

**检测方法**：
```bash
grep -ciE "(简介|介绍|什么是|what is|background|overview|^# .*入门|^# .*指南|^# .*介绍)" AGENTS.md
```
**修复**：删除所有教程内容，Agent 不需要学语言，它需要知道在这个项目中怎么做。

---

### AP-02: 目标缺失

**表现**：Agent 不知道自己在这个项目里的角色和核心目标。

**错误示例** ❌：
```
## 核心原则
- 写高质量的代码
- 注意测试覆盖
```

**正确示例** ✅：
```
## 核心原则
### R1: 优先实现核心业务逻辑
项目定位是一个数据处理管道，Agent 的职责是保证数据流的正确性和效率，
UI 和展示层的优化可以延后。

### R2: 测试覆盖核心路径
数据处理的关键路径（数据导入→清洗→转换→导出）必须有测试覆盖。
```

**检测方法**：通读核心原则，是否能回答"Agent 在这个项目中最重要的是什么？"
**修复**：在核心原则中添加与产品定位直接相关的 1-2 条原则。

---

### AP-03: 边界模糊

**表现**：只说了能做什么，没说什么不能做，或边界描述不具体。

**错误示例** ❌：
```
Agent 可以修改项目中的文件。
```

**正确示例** ✅：
```
### ❌ 禁止的操作
| 操作 | 原因 | 替代方案 |
|------|------|---------|
| 修改 deploy/ 下任何文件 | 直接关联生产环境 | 提交 PR 走审批流程 |
| 运行 .sql 文件 | 可能影响数据库 | 使用迁移脚本 |
| 删除 tests/ 下的文件 | 破坏测试基础设施 | 标记为废弃而非删除 |
```

**检测方法**：
```bash
grep -cE "(禁止|不允许|不能|不要)" AGENTS.md
# 结果应 ≥ 3
```
**修复**：添加显式的"禁止操作"章节。

---

### AP-04: 规则不可验证

**表现**：规则使用了无法判断是否被执行的语言。

**错误示例** ❌：
```
- 注意代码质量
- 尽可能提供高效实现
```

**正确示例** ✅：
```
- 提交前运行 `cargo test`，确保通过率 100%
- 新代码必须包含单元测试，覆盖率 ≥ 80%
- 移除未使用的导入和变量（clippy 零警告）
```

**检测方法**：随机选 3 条规则，是否能编程验证？
**修复**：将模糊描述改为可量化的条件。

---

### AP-05: 万能句式

**表现**：使用了放之四海而皆准的描述。

**错误示例** ❌：
```
- 遵循最佳实践
- 保持代码简洁
```

**正确示例** ✅：
```
- 遵循项目配置的 Prettier 和 ESLint 规则（见 .eslintrc.js）
- 函数不超过 50 行，超过时拆分
```

**检测方法**：检查是否有可以照搬到任何项目中都不违和的句子。
**修复**：将通用建议替换为项目特定的规则。

---

### AP-06: 越界承诺

**表现**：承诺了 Agent 能力范围外的事情。

**错误示例** ❌：
```
- Agent 可以修复所有 bug
- Agent 能保证 100% 的代码覆盖率
```

**正确示例** ✅：
```
- Agent 可以分析 bug 根因并提出修复方案，提交后需要开发者审查
- Agent 会运行测试并报告覆盖率，目标是 ≥ 80%
```

**检测方法**：标记所有"承诺性"的动词（保证、确保、一定）。
**修复**：将承诺性语言改为"提议—审查"模式。

---

### AP-07: 语气过软

**表现**：使用了"请考虑"、"如果可以的话"等可选化的表述。

**错误示例** ❌：
```
如果可以的话，请考虑运行测试。
```

**正确示例** ✅：
```
在提交前运行 `npm test`。如果有测试失败，先修复再提交。
```

**检测方法**：搜索"请"、"可以"、"如果可能"、"建议"。
**修复**：全部改为祈使句，以动词开头。

---

### AP-08: 冲突规则

**表现**：两条规则指向同一个行为但给出不同指示。

**错误示例** ❌：
```
规则 A: 所有代码提交必须经过 lint 检查
规则 B: 紧急修复可以直接提交，跳过 lint
```

**正确示例** ✅：
```
规则 A: 非紧急提交必须通过 lint 检查
规则 B: 紧急修复需要创建一个 `hotfix/` 分支，提交后再补 lint
```

**检测方法**：搜索是否有逻辑上矛盾的规则对。
**修复**：明确异常处理路径，而不是简单地让步。

## 二、结构反模式（5 种）

### AP-09: 没有门禁

**表现**：没有 `<HARD-GATE>` 或 `<GATE>` 标记，Agent 可以随意行事。

**检测方法**：
```bash
grep -c "<HARD-GATE>" AGENTS.md
# 结果应 ≥ 1
```
**修复**：在关键位置添加门禁标记。

---

### AP-10: 过长的入门介绍

**表现**：文件开头用 500+ 字的项目介绍，而不是直奔主题。

**检测方法**：`head -50 AGENTS.md`，检查前 50 行是否有实质性内容。
**修复**：前 50 行就应该包含核心原则或行为规则。

---

### AP-11: 缺少红线章节

**表现**：没有明确的"红线"或"禁止操作"章节。

**检测方法**：
```bash
grep -c "红线" AGENTS.md
grep -c "禁止" AGENTS.md
```
**修复**：添加红线章节，这是 AGENTS.md 最重要的部分之一。

---

### AP-12: 层级混乱

**表现**：章节嵌套深度 > 4 级（如 `####### 标题`），或者无规律。

**检测方法**：
```bash
awk '/^#{1,6} / {print NR, $0}' AGENTS.md | head -30
```
检查标题的层级是否有序递增。
**修复**：整理标题层级，确保不超过 3 级嵌套。

---

### AP-13: 重复内容

**表现**：相同的内容在两个不同的章节出现。

**检测方法**：通读全文，标记内容相似的段落。
**修复**：合并重复内容，用链接引用到唯一来源。

---

## 三、质量反模式（3 种）

### AP-14: 路径不验证

**表现**：引用了不存在的文件路径。

**检测方法**：
```bash
grep -oP '\]\([^)]+' AGENTS.md | sed 's/\]\(//' | while read p; do
  [ -e "$p" ] || echo "⚠️ 路径不存在: $p"
done
```
**修复**：修正不存在的路径引用。

---

### AP-15: 不更新版本号

**表现**：修改内容后没有更新版本号。

**检测方法**：检查版本历史是否与最新修改匹配。
**修复**：每次修改后更新版本号。

---

### AP-16: 不维护日志

**表现**：版本历史表格为空或过时。

**检测方法**：检查末尾的版本历史章节。
**修复**：每次修改时同步更新版本历史。

---

## 四、维护反模式（2 种）

### AP-17: 描述过窄

**表现**：description 过于精确，只覆盖了少量场景，导致 Agent 应该触发时没触发。

**检测方法**：是否能列举 5 个以上的触发场景？
**修复**：增加触发场景和同义词。

---

### AP-18: 描述过宽

**表现**：description 太通用，Agent 在不该触发时触发了。

**检测方法**：是否能列举一些"不应该触发但可能误触发"的场景？
**修复**：添加排除条件（"但不要用于 XXX 场景"）。
