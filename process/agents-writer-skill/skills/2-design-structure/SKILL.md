# ② 结构设计与蓝图生成

## 任务目标

根据 [① 项目分析](../skills/1-analyze-project/SKILL.md) 的类型判定结果，设计 AGENTS.md 的**完整结构蓝图**，包括章节规划、字段定义、优先级排序和内容密度控制。

## 触发条件

当用户说以下任一话术时激活：
- "设计AGENTS.md结构"
- "应该包含哪些章节"
- "帮我规划文件结构"
- "从零开始设计"

## 前置条件

⚠️ **必须先完成** [① 项目分析](../skills/1-analyze-project/SKILL.md)，获得：
- 项目类型（A/B/C/D/E）
- 复杂度评分（0-100）
- 关键特征列表

## 操作步骤

### Step 1: 确定基础结构（基于类型）

根据 [① 分析](../skills/1-analyze-project/SKILL.md) 结果，选择对应类型的标准结构作为起点。

**示例**：
```
输入: Type C (评分 52), MoonBit 技能库
输出: 采用"框架技能型"标准结构（11个必需章节 + 3个可选章节）
```

### Step 2: 自定义调整（差异化设计）

基于项目独特需求，调整标准结构：

#### 调整决策树

```
需要调整吗？
    │
    ├── 否 → 使用标准结构 ✅
    │
    └── 是 → 检查调整原因:
              │
              ├── 有特殊技术栈？
              │   └→ 增加"技术栈专项"章节
              │
              ├── 有复杂的工作流？
              │   └→ 拆分"通用工作流"为多个子章节
              │
              ├── 有严格的合规要求？
              │   └→ 增加"合规清单"章节
              │
              └── 有多团队协作？
                  └→ 增加"团队分工"章节
```

**常见调整场景**：

| 调整原因 | 建议操作 | 行数影响 |
|---------|---------|---------|
| 多后端支持（Wasm/JS/Native） | 增加"后端特性差异"章节 | +20-30 行 |
| FFI 集成（C/JS 互操作） | 增加"FFI 规范"章节 | +15-25 行 |
| Monorepo 管理 | 增加"包管理策略"章节 | +15-20 行 |
| DSL / 宏系统 | 增加"元编程规范"章节 | +20-30 行 |
| 高安全要求 | 增加"安全审计清单"章节 | +15-25 行 |

### Step 3: 定义字段规范（精细化设计）

对每个章节定义**字段级规范**：

```markdown
## 章节名称: {chapter_name}

### 目的
{为什么需要这个章节，解决什么问题}

### 必需字段
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| {field_name} | text/table/list | Yes/No | {...} |

### 可选字段
| 字段名 | 类型 | 推荐度 | 说明 |
|--------|------|--------|------|
| {field_name} | text/table/list | ⭐⭐⭐⭐☆ | {...} |

### 内容密度
- 最小行数: X 行
- 推荐行数: X-X 行
- 最大行数: X 行（超过则考虑拆分）

### 示例片段
```yaml
{实际示例}
```
```

**示例：意图路由表的字段规范**

```markdown
## 章节名称: 意图路由表

### 目的
将用户的自然语言请求映射到具体的 Agent 行为，实现"说X→做Y"的自动化调度。

### 必需字段
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 用户意图 | text | Yes | 简短描述（2-6字） |
| 触发示例 | text | Yes | 1-3个真实用户话术 |
| 执行动作 | text/link | Yes | Agent 应该做什么 |

### 可选字段
| 字段名 | 类型 | 推荐度 | 说明 |
|--------|------|--------|------|
| 优先级 | enum | ⭐⭐⭐⭐☆ | P0/P1/P2/P3 |
| 条件判断 | text | ⭐⭐⭐☆☆ | 前置条件或分支逻辑 |
| 相关子技能 | link | ⭐⭐⭐⭐☆ | 指向详细指南 |

### 内容密度
- 最小行数: 10 行（3个任务）
- 推荐行数: 15-25 行（5-8个任务）
- 最大行数: 40 行（超过则拆分为子技能索引）
```

### Step 4: 优先级排序（信息架构）

按照**阅读频率 × 重要程度**排列章节顺序：

#### 标准优先级顺序（高频优先）

```
1. 前言区 ← Agent 启动时首先解析
2. 身份与角色 ← 建立 WHO 上下文
3. 触发条件 ← 判断是否激活
4. 意图路由表 ← 90% 时间的入口
5. 通用工作流 ← 执行主流程
6. [核心业务章节...] ← 按需深入
   ├─ 编码规范
   ├─ 常用命令
   ├─ 错误处理
   └─ ...
N. 注意事项 ← 最后的安全网
```

**优先级调整原则**：

| 原则 | 应用场景 |
|------|---------|
| **Happy Path First** | 将最常用的 3 个任务放在意图路由表最前面 |
| **渐进式披露** | 基础信息前置，高级细节后置 |
| **故障快速定位** | 错误处理章节放在中后部（不需要每次都看，但要容易找到） |
| **启动性能优化** | 前言区和身份章节保持精简（< 20 行合计） |

### Step 5: 输出结构蓝图（交付物）

生成完整的结构设计文档：

```markdown
# 🏗️ AGENTS.md 结构蓝图

## 元信息
- **项目**: {project_name}
- **类型**: Type {X}: {type_name}
- **复杂度评分**: XX/100
- **目标行数**: XXX-XXX 行
- **创建日期**: {date}

## 章节规划（共 X 章）

### Layer 1: 核心骨架（必读，~XX 行）

| # | 章节名称 | 优先级 | 预估行数 | 状态 |
|---|---------|--------|---------|------|
| 1 | 前言区 (YAML) | P0 | 5-8 行 | ✅ 标准模板 |
| 2 | 身份与角色 | P0 | X-X 行 | ⬜ 待编写 |
| ... | ... | ... | ... | ... |

### Layer 2: 操作指南（按需查阅，~XX 行）

| # | 章节名称 | 触发频率 | 预估行数 | 状态 |
|---|---------|---------|---------|------|
| X | 意图路由表 | 90% | X-X 行 | ⬜ 待编写 |
| ... | ... | ... | ... | ... |

### Layer 3: 参考资料（低频查阅，~XX 行）

| # | 章节名称 | 查阅频率 | 预估行数 | 状态 |
|---|---------|---------|---------|------|
| X | 附录/术语表 | 5% | X-X 行 | ⬜ 可选 |

## 字段规范速查

[每个核心章节的字段定义表格...]

## 调整记录

| 日期 | 调整内容 | 原因 |
|------|---------|------|
| {date} | 增加"FFI规范"章节 | 项目有大量JS互操作代码 |

## 下一步
→ [③ 编写内容](../skills/3-write-content/SKILL.md) 或
→ [④ 选择模板](../skills/4-select-template/SKILL.md)
```

## 常见问题

**Q: 章节太多怎么办？**

A: 如果某章节超过 **50 行**，考虑：
1. 拆分为子技能（放入 skills/ 目录）
2. 或移入 references/ 作为详细参考
3. 在主文件只保留摘要 + 链接

**Q: 必须严格按照类型模板吗？**

A: **不必须**。类型模板是**起点**不是**终点**。根据实际需求自由调整，但保持整体合理性。

**Q: 如何平衡完整性和简洁性？**

A: 使用**三层信息架构**：
- **Layer 1** (前 20%): 90% 场景够用
- **Layer 2** (中间 60%): 按需深入
- **Layer 3** (后 20%): 低频参考

## 注意事项

⚠️ **Token 预算意识**: 设计时要考虑最终文件的 Token 消耗。详见 [token-budget.md](../references/token-budget.md)

⚠️ **可演进性**: 第一版不必完美。采用**增量式设计**——先出 MVP（最小可行结构），再根据使用反馈迭代。

⚠️ **一致性**: 同一层级的章节应保持相似的粒度和深度，避免有的很详细有的很粗糙。
