# Agent 讨论与用户交互

> 本文档定义了决策记录系统的 Agent 讨论功能、动态更新机制和用户交互协议。

## Agent 讨论功能模块

### 主动发起讨论的时机

Agent 应在以下场景**主动发起**决策讨论：

| 场景 | 触发条件 | 示例 |
|------|---------|------|
| **🚀 项目启动** | 首次进入 DECISION 阶段 | "在开始写作前，我们需要先确认几个关键决策..." |
| **⚠️ 缺失关键信息** | PLAN 阶段发现 critical 决策未确认 | "我注意到还没确定编程语言，这会影响 Ch3 的代码示例..." |
| **🔄 依赖链就绪** | 前置决策已确认，可以讨论后续 | "既然确定了用 Python，接下来我们讨论 RAG 框架的选择..." |
| **❌ 检测到冲突** | 新需求与已有决策矛盾 | "您刚才要求增加 JavaScript 示例，但之前确认了只用 Python..." |
| **📌 里程碑节点** | 完成某篇/某阶段 | "Ch3 已完成，接下来写 Ch4 前需要确认向量数据库的选择..." |
| **👤 用户询问** | 用户主动问及某方面 | "关于技术栈的问题，让我为您梳理一下当前的选择..." |

### 讨论引导流程（6 步法）

```
Step 1: 准备 (PREPARE)
  │
  ├── 确定本次讨论的主题范围（单决策/批量决策）
  ├── 收集相关决策的当前状态
  ├── 准备背景信息和上下文
  └── 检查依赖关系，确定讨论顺序
  │
Step 2: 呈现 (PRESENT)
  │
  ├── 清晰陈述决策问题
  ├── 说明这个决策的重要性（Why it matters）
  ├── 展示可选方案及其优缺点
  └── 给出 Agent 的建议（带理由）
  │
Step 3: 引导 (GUIDE)
  │
  ├── 使用 AskUserQuestion 工具呈现选项
  ├── 支持用户选择"Other"自定义输入
  ├── 对用户的犹豫提供额外解释
  └── 允许用户跳过（标记为 deferred）
  │
Step 4: 确认 (CONFIRM)
  │
  ├── 复述用户的选择
  ├── 说明选择的影响范围
  ├── 询问是否需要调整相关决策
  └── 获得最终确认
  │
Step 5: 记录 (RECORD)
  │
  ├── 更新决策状态为 confirmed
  ├── 记录 current_value 和 reasoning
  ├── 写入 change_history（如有变更）
  ├── 更新 discussion_sessions 日志
  └── 触发持久化到 JSON
  │
Step 6: 后续 (FOLLOW-UP)
  │
  ├── 检查是否有依赖此决策的其他待讨论项
  ├── 如有 → 自动加入下一轮讨论队列
  ├── 如无 → 总结本次讨论成果
  └── 返回主工作流（PLAN/RESEARCH/WRITE...）
```

### 讨论脚本模板

```markdown

### 背景
{为什么需要做这个决策 + 不做的影响}

### 当前状态
- **状态**: pending / confirmed({value}) / changed
- **默认值**: {default_value}
- **优先级**: critical/high/medium/low
- **影响范围**: {impact_scope}

### 可选方案

| 方案 | 推荐度 | 说明 | 适用场景 |
|------|--------|------|---------|
| A: {option1} | ⭐⭐⭐⭐⭐ | {description} | {when_to_use} |
| B: {option2} | ⭐⭐⭐⭐ | {description} | {when_to_use} |
| C: {option3} | ⭐⭐⭐ | {description} | {when_to_use} |

### 💡 Agent 建议
{基于最佳实践和项目特点的建议 + 理由}

### 请选择
[请等待用户回应...]
```

**批量决策讨论模板**（适用于关联紧密的决策组）：

```markdown
## 📋 批量决策讨论：{主题组名称}

本组包含 N 个相互关联的决策，建议一起讨论以确保一致性。

### 决策清单

2. **{决策2}** (high) - 当前: confirmed(Python)
3. **{决策3}** (high) - 当前: pending [依赖决策1]

### 讨论顺序建议
按依赖关系：决策1 → 决策2 → 决策3

---

### 开始第 1 个决策：{决策1}
{同单决策模板}

---

[完成第 1 个后自动进入第 2 个...]
```

### 智能推荐算法

Agent 应根据以下因素给出**个性化推荐**：

| 因素 | 权重 | 判断逻辑 |
|------|------|---------|
| 项目特征 | 30% | RAG 教程 → Python + LangChain 是主流选择 |
| 目标读者 | 25% | 初学者多 → 选 ChromaDB（简单）；高级开发者多 → 选 FAISS（性能） |
| 行业趋势 | 20% | 2025-2026 年数据 → 本地模型部署是热点 |
| 已有决策 | 15% | 选了 Python → 排除 JavaScript 专属框架 |
| 用户历史 | 10% | 之前的偏好倾向 |

**推荐格式**：
```
💡 我推荐选择 **Python** 作为主要编程语言，原因如下：
   1. AI/ML 领域绝对主流（90%+ 的 RAG 项目使用 Python）
   2. 生态丰富：LangChain/LlamaIndex/PyTorch 都首选 Python
   3. 降低读者门槛：大多数目标读者的技术栈是 Python

   当然，如果您有特殊考虑（如面向前端开发者），我们可以讨论其他选项。
```

## 动态更新功能

### 支持的操作类型

| 操作 | 触发方式 | Agent 行为 | 示例 |
|------|---------|----------|------|
| **✅ 确认决策** | 用户在讨论中选择 | 更新 status → confirmed, 记录 value | "好的，已确认使用 Python" |
| **✏️ 修改决策** | 用户提出变更请求 | 更新 value, 记录 change_history | "把框架从 LangChain 改为 LlamaIndex" |
| **➕ 补充决策** | 发现新的决策点 | 创建新决策项, 加入 decisions | "需要增加一个关于测试框架的决策" |
| **⏸️ 推迟决策** | 用户暂时无法决定 | status → deferred, 设置提醒 | "这个后面再决定" |
| **❌ 取消决策** | 决策不再需要 | status → cancelled, 归档 | "不需要考虑这个了" |
| **🔄 回滚决策** | 恢复到历史版本 | 从 change_history 恢复 | "改回到之前的选择" |

### 修改决策的标准流程

```
用户提出："我想把编程语言改成 JavaScript"

Step 1: Agent 接收请求
  │
  ├── 识别目标决策: tech_primary_language
  ├── 当前值: Python (confirmed at 2026-05-20)
  │
Step 2: 影响评估
  │
  ├── 检查依赖此决策的下游决策:
  │   ├─ tech_framework_preference → 可能需调整（JS 生态不同）
  │   ├─ content_example_style → 代码示例需重写
  │   └─ 影响范围: Ch3-Ch11（已写完的章节可能需修改）
  │
  ├── 生成影响报告:
  │   ⚠️ 变更影响评估
  │   ─────────────────────────────────────────
  │   直接影响:
  │   - 4 个下游决策可能需要调整
  │   - 已完成的 3 个章节(Ch3-Ch5)代码示例需改为 JS
  │
  │   建议:
  │   - 如果只影响后续章节(Ch6+) → 可以改
  │   - 如果需要重写已完成章节 → 建议慎重考虑
  │
Step 3: 呈现影响并确认意图
  │
  ├── 展示影响报告
  ├── 询问:"确认要修改吗？这将影响..."
  ├── 选项: [确认修改] [取消] [仅影响后续章节]
  │
Step 4: 执行修改（用户确认后）
  │
  ├── 记录旧值到 change_history:
  │   {
  │     from_value: "python",
  │     to_value: "javascript",
  │     changed_at: "2026-05-25T14:00:00",
  │     reason: "用户要求改为 JavaScript 以面向前端开发者",
  │     changed_by: "user"
  │   }
  │
  ├── 更新 current_value: "javascript"
  ├── 更新 status: "changed"
  ├── 更新 last_updated
  │
Step 5: 级联处理
  │
  ├── 标记受影响的下游决策为 needs_review
  ├── 生成待办事项:
  │   - [ ] 重新讨论 tech_framework_preference
  │   - [ ] 检查 Ch3-Ch5 是否需要修改
  │   - [ ] 更新资料搜索关键词（加 JavaScript 相关）
  │
Step 6: 持久化 & 通知
  │
  ├── 保存到 JSON 文件
  ├── 更新 version_history
  ├── 向用户展示变更摘要
  └── 建议下一步行动
```

### 版本管理机制

**轻量级版本控制**（非 Git，而是决策自身的版本追踪）：

```json
{
  "version_history": [
    {
      "version": "v0.1.0",
      "timestamp": "2026-05-25T00:00:00+08:00",
      "changes_summary": "初始版本：创建决策记录模板",
      "changed_decisions": ["全部20项"],
      "snapshot_hash": "sha256:abc123..."
    },
    {
      "version": "v0.2.0",
      "timestamp": "2026-05-25T10:30:00+08:00",
      "changes_summary": "确认了 5 个 critical 决策（语言/编程语言/用户画像/深度/质量标准）",
      "changed_decisions": [
        "basic_lang_primary",
        "tech_primary_language",
        "content_target_audience",
        "content_depth_level",
        "project_quality_standard"
      ],
      "snapshot_hash": "sha256:def456..."
    }
  ]
}
```

**版本号规则**：

| 变更类型 | 版本增量 | 示例 |
|---------|---------|------|
| 新增决策项 | minor+1 | v0.1.0 → v0.2.0 |
| 确认 N 个决策 | patch+N | v0.2.0 → v0.2.5 |
| 修改已确认决策 | minor+1 | v0.2.5 → v0.3.0 |
| 批量变更(>5 项) | major+1 | v0.3.0 → v1.0.0 |

**快照功能**：

每次重大变更（major/minor 版本升级）时，保存完整快照：
```
snapshots/
  └── v0.1.0-decision-record-20260525.json  ← 完整副本
  └── v0.2.0-decision-record-20260525.json
```

### 补充新决策的流程

当发现预定义的 20 个决策项不足以覆盖新场景时：

```
检测到新决策点（如："是否需要包含 Docker 部署示例？"）

Step 1: 评估必要性
  │
  ├── 是否真的需要一个独立决策？
  │   ├─ 是 → 继续创建
  │   └─ 否 → 作为现有决策的一个选项或备注即可
  │
Step 2: 设计决策项
  │
  ├── 选择合适的分类（基础配置/技术选型/...）
  ├── 定义 ID: `{category}_{descriptive_name}`
  ├── 编写 question、options、default_value
  ├── 确定 priority 和 dependencies
  │
Step 3: 注册到决策记录
  │
  ├── 添加到 decisions 对象
  ├── 更新 metadata.total_decisions
  ├── 初始状态: pending
  │
Step 4: 触发讨论（如需要立即决定）
  │
  ├─ priority=critical/high → 立即发起讨论
  └─ priority=medium/low → 加入待讨论队列

示例输出：
  ```json
  {
    "tech_include_docker": {
      "id": "tech_include_docker",
      "category": "技术选型",
      "question": "是否在教程中包含 Docker 容器化部署示例？",
      "type": "boolean",
      "default_value": true,
      "status": "pending",
      "priority": "medium",
      "impact_scope": ["Ch9 生产部署篇"],
      "created_at": "2026-05-25T15:00:00+08:00"
    }
  }
  ```
```

## 用户讨论交互流程

### 信息呈现规范

**何时展示决策状态给用户**：

| 场景 | 展示方式 | 详细程度 |
|------|---------|---------|
| 用户主动询问 | 完整决策报告 | 所有字段 |
| 章节开始前 | 相关决策摘要 | 仅影响本章的 |
| 检测到冲突时 | 对比视图 | 涉及冲突的决策 |
| 里程碑回顾 | 进度仪表盘 | 统计+关键决策 |
| 交付总结 | 决策版本快照 | 已确认的全部 |

**决策状态可视化模板**：

```markdown
## 📊 决策状态总览

**最后更新**: 2026-05-25 14:00
**进度**: 12/20 已确认 (60%) | 5 个讨论中 | 3 个待讨论

### ✅ 已确认决策 (12)

| 分类 | 决策 | 值 | 确认时间 |
|------|------|-----|---------|
| 基础配置 | 主要语言 | 简体中文 | 05-25 09:00 |
| 技术选型 | 编程语言 | Python | 05-25 09:30 |
| ... | ... | ... | ... |

### 🔄 讨论中 (5)

| 决策 | 问题 | 当前进展 |
|------|------|---------|
| 向量数据库 | 选择哪个向量数据库？ | 用户倾向于 FAISS，正在比较性能数据 |

### ⏳ 待讨论 (3)

| 决策 | 优先级 | 建议讨论时机 |
|------|--------|-------------|
| 图文比例 | medium | Ch1 规划前 |
| 反馈机制 | low | 全书初稿完成后 |
```

### 冲突检测与处理

**冲突类型**：

| 类型 | 检测信号 | 示例 | 处理方式 |
|------|---------|------|---------|
| **直接矛盾** | 新输入与已确认值冲突 | 已确认 Python，用户要求 JS 示例 | 触发修改流程 |
| **隐含矛盾** | 多个决策组合后不合理 | 选了初学者+专家级深度 | 提示组合风险，建议调整 |
| **过时决策** | 外部环境变化导致决策失效 | 某框架停止维护 | 建议重新评估 |
| **依赖断裂** | 前置决策变更未通知下游 | 改了语言但没更新框架选择 | 自动标记 needs_review |

**冲突处理流程**：

```
检测到冲突
  │
  ├── 识别冲突类型
  ├── 收集相关信息（两个决策的当前值、确认时间、原因）
  │
  ├── 生成冲突报告:
  │   ⚠️ 决策冲突检测
  │   ─────────────────────────────────────
  │   ┆
  │   冲突类型: 直接矛盾
  │   ┆
  │   决策 A: tech_primary_language
  │   - 已确认值: Python (2026-05-20 by user)
  │   - 原因: AI/ML 领域主流
  │   ┆
  │   决策 B: 用户新请求
  │   - 请求值: JavaScript
  │   - 原因: 面向前端开发者
  │   ┆
  │   影响范围: Ch3-Ch11 的代码示例
  │
  ├── 呈现给用户:
  │   ├── 说明冲突点
  │   ├── 展示双方理由
  │   ├── 分析影响范围
  │   └── 提供解决方案:
  │       ├─ 方案1: 修改决策 A（改用 JS）
  │       ├─ 方案2: 维持决策 A（保持 Python，JS 作为补充）
  │       ├─ 方案3: 分章节处理（前期 Python，后期介绍 JS）
  │       └─ 方案4: 推迟决定（先收集更多信息）
  │
  └── 用户选择后执行对应流程
```

### 变更请求处理

**用户发起变更的标准对话模式**：

```
用户: "我想把质量标准改成草稿级"

Agent 响应:
  1. 确认理解: "您希望将 project_quality_standard 从 '标准级' 改为 '草稿级'，对吗？"

  2. 影响评估:
     变更影响:
     - REVIEW 阶段的通过门槛降低
     - 可能允许更多不完美内容通过
     - 适合快速迭代场景

     当前已基于'标准级'完成了 Ch3-Ch5
     是否需要重新审查这些章节？

  3. 选项呈现:
     [确认修改（全局生效）]
     [仅影响后续章节（Ch6 起）]
     [取消，保持现状]

  4. 执行 & 记录

  5. 后续建议:
     "已修改为草稿级。建议：
      - Ch3-Ch5 可选择性重新 Review（非强制）
      - 从 Ch6 开始正式应用新标准
      - 如需回归严格模式，随时告诉我"
```
