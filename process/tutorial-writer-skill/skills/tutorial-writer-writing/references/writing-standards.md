# 写作规范详细版

> 本文档是 Tutorial Writer 技能的写作规范参考，定义了教程撰写的强制性规则、推荐惯例和格式标准。

---

## 1. 强制规则（R1-R10）

### R1: 使用目标语言写作

- 全篇使用目标语言撰写（如中文）
- 专有术语首次出现时保留英文原文并括号注明中文
- 示例：`RAG（Retrieval-Augmented Generation，检索增强生成）`
- **检测方式**：通读全文检查

### R2: 不添加注释性文字

- 教程内容中不包含 `# 注释`、`// TODO`、`待补充` 等占位符
- 不添加面向开发者的元说明文字
- 内容即交付物，不需要额外的"解释性注释"
- **检测方式**：全文搜索 `# `、`TODO`、`待补充`、`占位符`

### R3: 关键数据标注来源

- 所有非通用知识的数据、统计、引用必须标注来源
- 格式：`（来源: [描述](路径)）` 或 `（来源: 来源名 年份）`
- 示例：`（来源: Gartner 2025）`、`（来源: [幻觉率基准](reference/01-大模型基础/01-幻觉率基准与模型排行.md)）`
- **检测方式**：逐条检查文中的数字和论断

### R4: 代码块标注语言

- 每个代码块必须标注语言类型
- 格式：```python ```bash ```yaml
- 常见标注：python / bash / yaml / json / sql / markdown
- **检测方式**：视觉检查所有代码块

### R5: 内容对齐大纲/规划

- 每章内容必须匹配大纲或规划中的预定结构
- 小节标题应与大纲一致（允许微调措辞）
- 不允许自由发挥偏离主题的内容
- **检测方式**：对照大纲/规划逐节对比

### R6: 资料归档到指定目录

- 所有参考资料归档到项目约定的资料目录（如 reference/）
- 按主题分类存放，不允许散落在各处
- 新增资料需更新资源索引文件
- **检测方式**：检查文件位置是否符合目录规范

### R7: 跨章技术栈一致性（★ v2.1 新增）

> **来源问题**：RAG 教程优化中发现 Ch3 主张"全栈原生实现"，Ch5 却使用 LangGraph，导致读者困惑。

- 全书技术栈选型必须在 DECISION 阶段统一确认，后续章节不得擅自偏离
- 如果需要在特定章节引入新框架/工具，必须在该章节明确说明"本章引入 XX 作为补充，完整技术栈选型见第 X 章"
- HTTP 库、消息类型、数据格式等跨章共用的 API 风格必须全文统一
- **检测方式**：Grep 搜索全书中的框架引用，检查是否与决策记录一致

### R8: 跨章数据结构兼容性（★ v2.1 新增）

> **来源问题**：Ch4 输出 `{"text": str}` 格式，Ch5 使用 `chunk.page_content`（LangChain 格式），导致代码无法衔接。

- 当数据在章节间流动时（如上一章的输出是下一章的输入），必须确保数据结构格式一致
- 在涉及数据流转的章节开头，必须明确声明输入数据的格式（推荐使用 blockquote 标注）
- 如果数据格式发生变化（如重构），必须同步更新所有引用该数据的后续章节
- **检测方式**：检查相邻章节间的数据结构引用是否一致

### R9: 代码质量标准（★ v2.1 新增）

> **来源问题**：多章代码存在分隔符逻辑错误、同步/异步混用、死代码、硬编码假数据等问题。

- 代码示例中的关键逻辑必须经过人工审查，不能有明显的逻辑错误
- 异步上下文中禁止使用同步 HTTP 调用（如 `httpx.post` 而非 `await client.post`）
- 代码中不应存在未使用的函数、变量或 import（死代码）
- 如果代码使用模拟/硬编码数据，必须在注释中明确标注 `# 示例数据` 或 `# 生产环境需替换为实际实现`
- 分隔符、正则表达式等容易出错的细节必须经过验证
- **检测方式**：逐段审查代码逻辑，特别关注数据流转、异步调用、边界条件

### R10: 网页友好内容设计（★ v2.2 新增 → v3.1 增强）

> **核心理念**：教程的网页版不是 Markdown 的静态投影，而是可交互、可探索、可参与的学习体验空间。撰写时需主动设计交互触点。

#### 内容交互化检查清单

撰写每小节时，对照以下清单检查可增强之处：

| 内容类型 | 静态写法（❌） | 网页增强（✅） |
|---------|--------------|--------------|
| 架构图 | 一张截图 | Mermaid 流程图 + 可点击区域高亮 |
| 代码示例 | 纯文本代码块 | 可运行沙盒（Monaco Editor + Pyodide） |
| 性能对比 | 数据表格 | Chart.js 雷达图/条形图 + 悬停显示详情 |
| 步骤流程 | 有序列表 | CSS 滚动触发动画逐步揭示 |
| 知识检验 | 无 | 选择题卡片组件 |
| 技术演进 | 文字描述 | 可拖拽时间轴 |
| 复杂概念 | 长篇文字 | 3D 可视化可旋转模型 |

#### 嵌入组件标记规范

在 Markdown 中插入交互组件时，遵循以下规范：

```markdown
<!-- 交互组件占位标记规范 -->
<!-- 格式：<!--交互:类型:ID:说明--> -->
<!-- 写在 book/ 中的章节文件里，同步到 web/ 后 JS 脚本解析执行 -->

<!-- 代码沙盒（JS 文件在 web/js/） -->
<!--交互:playground:pg-01:向量检索示例-->

<!-- 交互测验（JS 文件在 web/js/） -->
<!--交互:quiz:qz-01:RRF 作用检验-->

<!-- 3D 可视化（JS 文件在 web/js/3d/） -->
<!--交互:3d:3d-01:RAG 架构层次-->

<!-- 动画图表（JS 文件在 web/js/） -->
<!--交互:chart:ct-01:架构性能雷达图-->
```

这些标记在 web-publishing 阶段的 JS 渲染脚本中会被解析为实际的交互组件。对应的 JS/CSS 文件直接维护在 `web/js/` 和 `web/css/` 中。

#### 交互组件内容编写原则

- **组件自解释**：每个交互组件旁的文本应说明"这是什么"和"如何操作"
- **渐进式披露**：先展示基础内容 → 再提供交互操作 → 最后给出深度探索入口
- **保持 Markdown 可读**：确保纯 Markdown 视图下内容完整可理解，交互组件仅作为增强
- **fallback 优先**：始终提供静态内容的降级方案（图片、文字描述）

> **来源问题**：教程中的图片、Mermaid 图表、跨章链接在本地 Markdown 查看器（如 Typora）中正常，但在网页构建后出现路径失效或渲染失败。

- **内容编辑**：章节内容编辑在 `book/` 中进行，完成后同步到 `web/chapters/`
- **图片路径**：Markdown 文件在 `book/` 时引用绝对项目路径；同步到 `web/chapters/` 后引用相对于 `web/` 的路径（`../assets/images/xxx.png`）
- **图片存储**：图片和静态资源存储在 `web/assets/images/`，禁止引用 `book/` 内的图片（book/ 不发布）
- **图片格式**：优先使用 WebP 格式（压缩率高），兼容 Fallback 提供 PNG。分辨率建议 800-1200px 宽
- **Mermaid 图表**：必须经过语法验证（可使用 https://mermaid.live 在线调试），节点数不超过 15 个，避免过于复杂的布局
- **跨章节链接**：使用相对于 `web/` 的路径（`../chapters/chXX-xxx.md`），网页构建后 MkDocs 自动转换为站内 URL
- **交互组件**：自定义 JS 放在 `web/js/`，CSS 放在 `web/css/`，在 `web/` 下的 Markdown 中通过 `<div>` 容器嵌入。通过 `DOMContentLoaded` 事件初始化，避免与 MkDocs 的 instant loading 冲突
- **参考资料**：`reference/` 仅在开发时查阅，不发布到网页。所有必现资源必须放入 `web/`
- **表格宽度**：列数不超过 5 列，内容不超过 30 字符/格，避免窄屏滚动
- **代码行宽**：代码示例单行不超过 80 字符，超出需换行或简写
- **文章内引用**：`book/` 中的 Markdown 使用 `（来源: [reference/...]）` 格式；同步到 `web/` 后，引用内容必须转为 `web/` 下可访问的资源
- **检测方式**：`mkdocs build` 构建后检查 `site/` 目录中资源文件是否存在，浏览器预览无 404

## 2. 推荐惯例（S1-S5）

### S1: 先讲原理再给实践

每节内容遵循 "Why → What → How" 的顺序：

```
原理阐述（为什么需要这个技术/方法）
  ↓
概念解释（它是什么）
  ↓
实践操作（怎么做 / 代码示例）
  ↓
效果验证（如何确认做对了）
```

**反例**（直接给代码不讲原因）：
> 安装依赖：`pip install chromadb`
> 加载数据：...

**正例**（先讲为什么）：
> Chroma 是一个轻量级向量数据库，适合快速原型开发。它的核心优势是 API 极简——5 行代码即可完成向量存储和检索。
>
> 安装依赖：
> ```bash
> pip install chromadb
> ```

### S2: 复杂概念用比喻说明

当遇到抽象技术概念时，用生活化的比喻降低理解门槛：

| 抽象概念 | 比喻 | 说明 |
|---------|------|------|
| RAG | 开卷考试 | LLM 是考生，外部知识库是课本 |
| Embedding | 语义坐标 | 把文本映射到多维空间中的坐标点 |
| 向量数据库 | 语义图书馆 | 按"意思相近"而非"字面相同"来排列书籍 |
| HNSW 索引 | 高速公路网 | 多层互通的高速公路系统，快速到达目的地 |
| RRF 融合排序 | 多评委打分汇总 | 多个检索器各自排名后加权融合 |

### S3: 表格对比替代长段落

当需要比较多个方案、策略、指标时，优先使用表格：

**反例**（长段落）：
> LangChain 是最主流的框架，GitHub 星数超过 11.9 万。LlamaIndex 是 RAG 领域的垂直标杆，星数超过 4.4 万。LangChain 的优势在于生态丰富... LlamaIndex 的优势在于数据聚焦...

**正例**（表格）：

| 对比维度 | LangChain | LlamaIndex |
|---------|-----------|------------|
| 核心赛道 | 通用型 LLM 应用编排 | RAG 与检索增强专项 |
| GitHub 星数 | 119K+ | 44K+ |
| RAG 能力 | 8/10 | 10/10 |

### S4: 引用参考文件编号方便追溯

在引用参考资料时，使用简洁的编号引用格式：

- `（来源: [幻觉率基准]）` — 引用 reference 目录下的具体文件
- `（来源: Gartner 2025）` — 引用外部权威报告
- `（来源: NAACL 2025 arXiv:2410.13070）` — 引用学术论文

避免过长的完整路径引用，保持正文整洁。

### S5: 技术深度六维模型（★ v2.1 新增）

> **来源问题**：用户反馈"技术介绍要有深度，让读者知其然，知其所以然"。优化前多章技术介绍仅停留在"怎么用"，缺少"为什么"和"来龙去脉"。

每个核心技术点应覆盖以下六个维度（不必每点都全覆盖，但核心概念至少覆盖 4/6）：

| # | 维度 | 说明 | 示例问题 |
|---|------|------|---------|
| ① | **概念定义** | 这个技术/方法是什么？ | "MMR 是一种在相关性和多样性之间取得平衡的排序算法" |
| ② | **核心原理** | 它的底层机制是什么？公式/算法是什么？ | "MMR 通过 λ 参数权衡相关性得分和新颖性得分" |
| ③ | **解决问题** | 为什么需要它？解决了什么痛点？ | "语义检索返回的 Top-K 结果往往高度相似，浪费上下文窗口" |
| ④ | **优缺点** | 有什么优势和局限性？ | "优点：简单高效；缺点：需要额外的 embedding 计算" |
| ⑤ | **发展历程** | 起源于哪里？当前发展到什么阶段？ | "1998 年 Carbonell 提出，2023 年后在 RAG 场景中广泛采用" |
| ⑥ | **注意事项** | 使用时有哪些常见陷阱？生产环境要注意什么？ | "λ 参数需要根据场景调优，0.5-0.7 是常见起点" |

**深度检查清单**（REVIEW 阶段使用）：

```
对每个核心技术点检查：
  ├─ 是否解释了"为什么需要"（不只说"怎么做"）？
  ├─ 是否说明了核心原理/机制？
  ├─ 是否给出了优缺点对比？
  ├─ 是否提到了常见陷阱或注意事项？
  └─ 是否有至少一个具体案例或数据支撑？
```

**反例**（仅说"怎么做"）：
> MMR 通过 `collection.query(..., where_document=...)` 实现去重。

**正例**（六维覆盖）：
> **为什么需要 MMR**：语义检索返回的 Top-K 结果往往高度相似——例如检索"Python 异步编程"可能返回 5 篇几乎相同的文章，浪费宝贵的上下文窗口。
>
> **核心原理**：MMR（最大边际相关性）通过 λ 参数在相关性和多样性之间取得平衡：`score = λ × sim(d, q) - (1-λ) × max(sim(d, d'))`。λ 越大越偏向相关性，越小越偏向多样性。
>
> **注意事项**：λ=0.5-0.7 是常见起点。高精度场景（法律、医疗）建议 λ≥0.7，探索性场景（研究助手）建议 λ≤0.5。

## 3. 格式要求详解

### 3.1 标题层级

```
# 第X章 章节标题          ← Level 1: 仅用于章节标题
## X.Y 小节标题           ← Level 2: 主要小节
### X.Y.Z 子小节标题      ← Level 3: 细分知识点
#### 极少使用             ← Level 4: 仅在必要时使用
```

**原则**：层级不超过 4 级。如果需要第 4 级，考虑是否应该拆分为独立小节。

### 3.2 代码块规范

```python
# 必须标注语言
def example_function():
    """代码应有实际意义，非伪代码"""
    result = compute_something()
    return result
```

**代码质量要求**：
- 代码应可直接运行（或明确标注为伪代码/片段）
- 关键行应有简要逻辑说明（在正文中，不是代码注释）
- 适配目标运行环境（如 Windows PowerShell vs Linux Bash）
- 变量命名清晰，符合语言惯例

### 3.3 列表规范

无序列表（`-`）用于：
- 要点列举
- 步骤说明（无严格顺序时）
- 特征列表

有序列表（`1.`）用于：
- 有严格顺序的操作步骤
- 编号列表
- 流程说明

### 3.4 表格规范

```markdown
| 列A | 列B | 列C |
|-----|-----|-----|
| 数据 | 数据 | 数据 |
| 数据 | 数据 | 数据 |
```

**要求**：
- 表头对齐，使用 `|` 分隔
- 列宽根据内容自适应
- 重要数值加粗或高亮
- 表格后有简要总结或结论

### 3.5 引用格式

```markdown
内联引用：（来源: [描述](路径)）
外部引用：（来源: 来源名 年份）
学术引用：（来源: 会议/期刊 论文ID）
```

## 4. 内容密度指南

### 4.1 信息密度目标

好的教程应在有限篇幅内传递最大信息量：

| 内容类型 | 信息密度 | 特征 |
|---------|---------|------|
| 高密度 | ★★★★★ | 每段都有实质性信息，无废话 |
| 中密度 | ★★★☆☆ | 有一定铺垫但不过度 |
| 低密度 | ★★☆☆☆ | 大量过渡性文字，信息稀疏 |

**目标**：整体维持中高密度。过渡性文字控制在每节 1-2 段以内。

### 4.2 避免低密度写法

| 低密度写法 | 高密度替代 |
|-----------|-----------|
| "正如我们在前面章节中讨论过的..." | 直接引用前文要点 |
| "这是一个非常重要的问题..." | 直接给出重要性和影响 |
| "让我们来看看..." | 直接展示内容 |
| "众所周知..." | 直接陈述事实或标注来源 |
| 三段式重复同一个观点 | 一段讲清，配表格总结 |

### 4.3 结构化表达优先

能用结构化方式表达的，不用纯文本：

| 表达需求 | 推荐格式 |
|---------|---------|
| 方案/工具对比 | 表格 |
| 操作步骤 | 有序列表 + 代码块 |
| 决策流程 | 文本流程图或 Mermaid |
| 层级关系 | 缩进列表或树形图 |
| 数值范围 | 表格 + 区间说明 |
