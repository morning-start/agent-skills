# 进度跟踪方法

> 本文档介绍项目规划的进度跟踪机制，包括里程碑管理、燃尽图、看板方法和健康度指标，帮助团队实时监控规划执行情况并及时调整。

---

## 一、为什么需要进度跟踪？

```
无跟踪的情况:
Q1 设定 OKR → Q1 中无人问津 → 季末发现只完成 30%
     ↓
结果: 目标落空，原因不明，无法改进

有跟踪的情况:
Q1 设定 OKR → 月度检查 (60%) → 调整策略 → 季末达成 75%
     ↓
结果: 虽未100%达成，但过程可控，有明确的学习点
```

**核心价值**: 进度跟踪不是监控，而是**及时发现偏差并纠正**。

---

## 二、4 种核心跟踪方法

### 方法 1: 里程碑跟踪

#### 适用场景
- 长期项目（季度/年度）
- 有明确交付节点的计划
- 需要向外部（客户/管理层）汇报的项目

#### 标准格式

```markdown
## 📍 里程碑状态总览

| 里程碑 | 计划日期 | 实际日期 | 状态 | 偏差 | 原因 |
|--------|---------|---------|------|------|------|
| M1: {name} | {plan} | {actual} | ✅/🟡/❌ | ±{N}d | - |
| M2: {name} | {plan} | {actual} | ⬜/🔄 | - | - |

### 健康度指标
- **准时率**: 按时完成的里程碑占比（目标 > 80%）
- **平均偏差**: 实际 vs 计划的平均天数差（目标 < ±5天）

#### 可视化建议
- 使用甘特图（Gantt Chart）展示时间线
- 用颜色标记状态：🟢按时 / 🟡延期 < 7天 / 🔴延期 ≥ 7天 / ⬜ 未开始

---

### 方法 2: 燃尽图 (Burn-down Chart)

#### 适用场景
- 敏捷开发（Sprint 级别）
- 任务量可预估的工作
- 需要预测是否能按期完成

#### 标准格式

```
剩余工作 (Story Points)
    │
 40 ─┼─────────────── 理想线
    │ ╲
 30 ─┤  ╲  ╲
    │   ╲  ╲  ╲
 20 ─┤   ╲  ╲  ╲  ╲  ← 实际线
    │     ╲  ╲  ╲  ╲
 10 ─┤       ╲  ╲  ╲  ╲  ╲
    │         ╲  ╲  ╲  ╲  ╲
  0 ─┴───┬───┬───┬───┬───┬→ Sprint End
        Day1 Day3 Day5 Day7 Day9 Today

解读:
• 实际线在理想线上方 → 进度落后 ❗
• 实际线在理想线下方 → 进度超前 ✅（注意质量）
• 两线接近平行 → 进度正常 ✅
```

#### 关键指标
- **Burn Rate**: 平均每天完成多少 Points？（用于预测未来）
- **Forecasted Completion**: 按当前速度，何时能完成？

---

### 方法 3: 看板 (Kanban)

#### 适用场景
- 持续流动的工作流（非固定迭代）
- 可视化任务状态和瓶颈
- 团队协作和透明化管理

#### 标准列配置

```
┌─────────┬─────────┬─────────┬─────────┐
│  Backlog │   To Do │  Doing │   Done  │
│  (待办)  │  (待开始)│ (进行中)│  (已完成)│
└─────────┴─────────┴─────────┴─────────┘
    ↑         ↑         ↑         ↑
  无限      WIP限制   WIP限制   无限制
  (通常)   (如 3个)  (如 3个)
```

#### WIP (Work In Progress) 限制
- **作用**: 限制同时进行的任务数量，暴露瓶颈
- **推荐值**: 团队人数 × 1.5 ~ 2（或根据历史数据调整）
- **效果**: 强制"完成后再开始新任务"，减少上下文切换

#### 看板健康度指标
- **Cycle Time**: 从 To Do 到 Done 的平均时间
- **Lead Time**: 从 Backlog 到 Done 的平均时间
- **Throughput**: 单位时间完成的任务数
- **Bottleneck**: 哪一列的任务积压最多？

---

### 方法 4: 健康度仪表盘 (Health Dashboard)

#### 适用场景
- 需要高层概览的管理者
- 定期（周/月）的团队健康检查
- 多维度综合评估

#### 核心维度

| 维度 | 指标 | 健康 (🟢) | 警告 (🟡) | 危险 (🔴) |
|------|------|-----------|-----------|-----------|
| **进度** | 计划完成率 | ≥ 85% | 70-85% | < 70% |
| **质量** | Bug 率比 | < 5% | 5-15% | > 15% |
| **范围** | 需求变更率 | < 10% | 10-25% | > 25% |
| **团队** | 人员稳定率 | > 90% | 80-90% | < 80% |
| **风险** | 未解决风险数 | 0-2 | 3-5 | > 5 |
| **技术债务** | 代码覆盖率 | > 80% | 60-80% | < 60% |

#### 红绿灯系统
```
🟢 绿灯 (All Green): 一切正常，继续执行
🟡 黄灯 (有 Yellow): 需关注，制定应对计划
🔴 红灯 (有 Red): 必须立即干预，升级处理
```

## 三、跟踪频率推荐

| 层级 | 方法 | 频率 | 参与人 |
|------|------|------|--------|
| **个人** | TODO 打勾 | 每日 | 自己 |
| **团队** | 站会 + 看板更新 | 每日/每两天 | 全员 |
| **Sprint** | 燃尽图 + Review | 每 Sprint | Scrum Team |
| **月度** | KR 数据更新 + 仪表盘 | 月初 | Team Lead + PM |
| **季度** | OKR 复盘 + Roadmap 更新 | 季末 | 全员 + Stakeholders |

## 四、跟踪工具推荐

| 工具类型 | 推荐工具 | 特点 |
|---------|---------|------|
| **轻量级** | Excel / Google Sheets / Notion 表格 | 灵活、免费、易上手 |
| **敏捷工具** | Jira / Trello / Asana / Linear | 专业、自动化、集成度高 |
| **看板专用** | Miro / Mural / LeanKit | 可视化好、适合远程团队 |
| **仪表盘** | Grafana / Metabase / 自建 Dashboard | 自动化数据采集、实时刷新 |

## 五、反模式：跟踪的常见错误

### ❌ 错误 1: 只跟踪不行动
**现象**: 每周开会记录进度，但对偏差不做任何调整  
**修正**: 建立"偏差阈值"，超过则触发应对方案

### ❌ 错误 2: 跟踪过细（微管理）
**现象**: 每小时问一次"做完了吗？"  
**修正**: 给予自主空间，只在关键节点检查（如每日站会、Sprint 结束）

### ❌ 错误 3: 只报喜不报忧
**现象**: Dashboard 全绿，实际一团糟  
**修正**: 鼓励透明文化，问题早暴露早解决

### ❌ 错误 4: 跟踪与目标脱节
**现象**: 任务都完成了，但 OKR/KR 没达成  
**修正**: 确保 TODO → Milestone → OKR 的对齐链路清晰
